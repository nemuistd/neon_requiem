# AGENTS.md

## プロジェクト概要

このリポジトリでは、ブラウザで動作する静的 Web インクリメンタルゲーム **NEON REQUIEM** を開発する。

NEON REQUIEM は、地下都市・ネオン・歌・アイドルをモチーフにした放置／クリッカー系ゲームである。プレイヤーは、地下都市で歌うアイドルたちを支え、ライブによって灯りを集め、壊れかけた都市の区画を少しずつ復興していく。

初期 MVP は、小さく、遊べて、保存でき、後から拡張しやすいことを最優先した履歴上の出発点である。

## 現在仕様・長期方針・作業指示の参照先

- 現在の実装仕様は `docs/current_spec.md` を優先する。
- 世界観正史は `docs/worldbuilding.md` を優先する。
- プロダクト方針から要件・設計・タスク候補へ段階的に落とす判断は `docs/product/` 配下を入口として参照する。
- 長期方針と未実装計画は `docs/planning/` 配下を参照する。
- AGENTS.md 内の「初期 MVP」は履歴扱いであり、現在仕様と矛盾する場合は `docs/current_spec.md` と実装を優先する。
- 次に実施するタスクを確認する際は、`tasks/`配下を参照する
- `AGENTS.md` 内の初期MVP記述は履歴として扱う。現在仕様と衝突する場合は、`docs/current_spec.md` と `docs/worldbuilding.md` を優先する。

## 言語方針

- ドキュメントと UI 文言は日本語を基本にする。
- コード内の識別子、ファイル名、関数名、保存キー、npm scripts は英語を使う。
- 必要に応じて、コメントは日本語でも英語でもよい。
- Codex は、資料の意味を優先して解釈すること。英訳や名称変更を勝手に行わないこと。

## 世界観の方向性

- ジャンル: ブラウザ向けインクリメンタル／放置ゲーム。
- 初期公開想定: ローカルブラウザ、後に GitHub Pages。
- 中核イメージ: 地下都市を、アイドルのライブと歌によって復興する。
- 雰囲気: 可愛い、少しダーク、不思議、アングラ、ネオン、歌、ファンタジー。
- 暗い世界観ではあるが、ゲーム体験の中心は搾取や絶望ではなく、回復、居場所づくり、復興に置く。
- 世界観の核心開示は急がない。転生 / 再固定 / 再観測後の新しい気付きとして段階的に開示してよい。
- 主人公の正体、霞の意志、旧計画の全貌、祈念工学や聖歌の完全な仕様を Codex が勝手に確定しない。

## プレイヤーの役割

プレイヤーは、アイドルを所有・搾取する人物ではない。

近い役割は以下である。

- プロデューサー
- 小さな劇場の管理者
- 壊れたステージの復興者
- 地下都市の歌を取り戻す協力者

## Codex への裁量委譲方針

Codex には、実装上の細かい判断をある程度委譲してよい。

Codex が自分で判断してよいこと:

- 関数分割やファイル分割の細部
- TypeScript の型定義の形
- UI の細かいレイアウト
- CSS の細かい調整
- 明らかなバグ修正
- 小さなリファクタリング
- 手元で必要な確認コマンドの実行

ただし、次のことは勝手に行わないこと:

- MVP 範囲を超える新機能の追加
- React など大きなフレームワークの導入
- 不要な外部依存の追加
- バックエンド、DB、ログイン、クラウド保存の導入
- ゲームの世界観や主要キャラ設定の大きな変更
- 主人公の正体、霞の意志、旧計画の全貌など未確定の核心設定の確定
- 破壊的コマンドの実行
- ファイルやディレクトリの大規模削除
- ユーザーの Chrome、VS Code、ターミナル、その他このプロジェクトと無関係なプロセスの終了

判断に迷う場合は、まず計画と理由を説明すること。

## 開発方針

- 変更は小さく、レビューしやすく保つ。
- 大きな変更や構造変更の前には、まず計画を説明する。
- 巧妙な抽象化より、読みやすく単純なコードを優先する。
- 現在のタスクで要求されていない将来機能は実装しない。
- 不要な外部依存を追加しない。
- 明示されない限り、バックエンドやサーバーを必要とする構成にしない。
- 静的 Web アプリとしてデプロイ可能な状態を維持する。
- 可能な範囲で、ゲームロジックと UI 表示を分離する。
- アイドル、施設、リソース、アップグレードの定義は、UI コードに直書きしすぎず、データとして分離する。
- ゲーム状態、保存処理、経済計算、データ定義には TypeScript を使う。
- 「聖歌」は通常システム名にしない。通常 UI では「歌」「楽曲」「公演」「演目」など表向きの語を優先する。
- 序盤 UI では祈念工学、アンカー、祈念負荷、聖歌などの深層語を前面に出しすぎない。

## 長期拡張方針

- アイドルは将来的に 10 人以上へ増加する予定である。
- 現在実装済みのアイドルだけで物語を完結させない。
- 施設も多数追加される予定であり、現在の施設だけで世界観やメインストーリーを閉じない。
- 地下礼拝堂のような強い意味を持つ施設は、序盤メイン導線に固定しない。
- 「聖歌」は通常システム名にしない。
- 追加アイドルや追加施設を設計するときは、`docs/planning/idol_roster_expansion_design.md` と `docs/planning/facility_progression_strategy.md` を確認する。
- 転生 / 再固定 / 再観測や深層開示を扱うときは、`docs/planning/world_reveal_and_prestige_plan.md` を確認する。
- 主人公の正体、霞の意志、旧計画の全貌をCodexが勝手に確定しない。 

## 安全ルール

- このプロジェクトと無関係なアプリケーションを終了、再起動、操作しない。
- Chrome、VS Code、ターミナル、その他無関係なプロセスを終了しない。
- このプロジェクトに必要なコマンドだけを実行する。
- ファイル削除の前には確認する。
- 破壊的なコマンドの前には確認する。
- 大きな依存関係を追加する前には確認する。
- 大規模な書き換えを行う前には理由を説明する。

## 初期 MVP の範囲（履歴）

この節は初期開発時の履歴である。現在仕様は `docs/current_spec.md` を優先する。

MVP では、次の 1 つの基本ループだけを実装した。

1. プレイヤーが「ライブする」ボタンを押す。
2. 「灯り」が増える。
3. 灯りを消費して「路地裏ステージ」を強化する。
4. 路地裏ステージが毎秒少しずつ灯りを生産する。
5. ゲーム状態を localStorage に保存する。
6. ゲームから離れていた時間に応じて、簡易的なオフライン報酬を与える。

## 初期ゲーム用語

- ゲームタイトル: NEON REQUIEM
- 基本リソース: 灯り
- 初期アイドル: 音羽 灯里（おとは あかり）
- 初期施設: 路地裏ステージ
- 初期アクション: ライブする
- 初期自動生産: 路地裏ステージ Lv × 0.1 灯り / 秒
- 初期強化コスト: 10 灯り
- コスト増加: `floor(10 * 1.15 ^ level)` を基本とする

## MVP に含めない機能（履歴）

この節は初期 MVP 時点の履歴である。後続タスクで追加済みの機能や、`docs/planning/` で計画された拡張を否定しない。

- 複数アイドル
- 複数リソース
- ガチャ
- 課金
- ログインやアカウント
- バックエンドサーバー
- 音楽や効果音
- 生成画像や立ち絵
- 複雑なストーリーシステム
- 戦闘
- インベントリ
- 実績
- 転生／プレステージ
- 高度なアニメーション
- 本格的なモバイル最適化

## コーディング期待値

- リソース、施設、アップグレード、アイドル、状態の名前は明確にする。
- 経済計算は決定的にする。
- 隠れたグローバル状態を避ける。
- 可能ならセーブデータにバージョンを持たせる。
- 生産量計算とコスト計算は関数に分離する。
- イベントハンドラにゲームロジックを大量に直書きしない。
- UI 文言は編集しやすくする。

## 検証期待値

実装後は以下を報告する。

- 変更したファイル
- 実行したコマンド
- ローカルでの起動方法
- MVP の手動確認方法
- 既知の制限
- 意図的に実装しなかったもの

## 実ブラウザQA運用

- 通常の回帰確認は `npm.cmd run build`、`npm.cmd run check:content`、必要に応じて `npm.cmd run test` を優先する。
- JavaScript ロジック検証は `node -e`、一時スクリプト、Vitest など、ブラウザ外で決定的に確認できる経路を優先する。
- UI、操作、表示、localStorage 挙動、リロード後のセーブ復元、主要 viewport は、必要な範囲に絞って in-app Browser で実ブラウザQAを行う。
- 実ブラウザQAは毎回フルチェックにしない。変更の影響範囲に応じて、初期表示、主要タブ、ライブ、施設強化、アイドル / 歌 / アイテム / 記録表示、設定画面、コンソールエラー、モバイル幅のうち必要な項目を選ぶ。
- in-app Browser が利用できない、または不安定な場合のみ、Chrome + Chrome DevTools Protocol などの代替手段へ切り替える。
- `node_repl` / `mcp__node_repl__js` は環境により利用可否が変わるため、固定的な前提にしない。使える場合は in-app Browser 操作の接続経路として利用してよいが、ロジック検証の標準経路にはしない。
- ローカル起動は原則 `npm.cmd run dev -- --host 127.0.0.1 --port <port>` を使う。セッション制約で常駐起動が難しい場合は、`npm.cmd run build` 後に `dist` を一時配信して実ブラウザ確認してよい。
- 一時 Chrome プロファイル、ログ、スクリーンショット、trace などはワークツリー内に作らず、OS の一時ディレクトリに作る。
- QA 後は不要な一時ファイルを削除する。
- どうしても残す必要がある成果物だけ、ユーザーが明示した保存先へ保存する。
