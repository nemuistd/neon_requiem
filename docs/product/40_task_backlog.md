# Task Backlog

## 位置づけ

この文書は、十分に具体化できた項目だけを、実行可能なタスク候補として置く場所である。

ここに書かれた項目は、ただちにゲーム実装を行う指示ではない。実装に進む時は、対象タスクを1つ選び、関連する上位文書、現在仕様、世界観正史、実装済みコードを確認してから着手する。

完了済みタスクの詳細な作業分担や古い実行計画は `docs/product/archive/` を参照する。現在仕様は `docs/current_spec.md` を優先する。

## バックログ投入条件

`40_task_backlog.md` に入れるタスクは、次を満たす必要がある。

- 目的が明確である。
- 対象範囲がレビューしやすい大きさである。
- 依存関係と推奨実行順が分かる。
- 並列可否と write scope が分かる。
- 受け入れ条件が確認可能である。
- 確認方法が、手動QA、コンテンツ検証、テスト、ビルドのいずれかに落ちている。
- [Product Charter](00_product_charter.md) と [Experience Requirements](10_product_requirements.md) に反していない。
- 実装済み仕様や正史に影響する場合は、同期先が明記されている。

## 未整理メモ / 要検討

2026-05-25 時点のプレイ所感メモ。ここにある項目は、まだ単独実装タスクではなく、次に UI 導線、廻後体験、バランスを見直す時の検討材料として扱う。

### 全体 UI 整理

- 廻関連の情報は、長期的には専用タブとして増やすより、別UIまたは既存導線内の整理で扱う方が収まりがよい可能性がある。
- 廻の清算は、通常のタブ内パネルではなく、他の情報へ目移りしない専用画面へ遷移して行う形を優先候補として検討する。
- 現在の廻タブは最小実装の確認場所としては有効だが、廻後ダッシュボード、追記索引、バフ確認、清算導線を今後も同じタブへ積み続けるかは再検討する。

### アイドル導線

- 左側の注目アイドル選択から未加入アイドルをクリックした場合も、そのアイドルへ声をかけて加入できる導線は最小実装済み。詳細パネルから交流イベントとアイドル由来記録を読む最小導線も実装済み。
- アイドル加入と注目切り替えの入口を左メニューに寄せるなら、アイドル詳細やアイドルタブは、交流増加で得た記録、交流イベント、読み返し要素を確認する場所として整理する案を検討する。
- アイドルとの交流によって解放される記録や短い読み物は、記録タブへ混ぜるだけでなく、アイドルタブまたは左のアイドル詳細から開ける場所に置く案を検討する。
- 表示は別ブラウザウィンドウではなく、現在の画面内で開閉できるポップアップ、詳細パネル、またはドロワーに近い形を優先候補にする。
- アイドル詳細を開いて読む形にすると、通常交流イベントや薄明の記憶イベントを追加しやすい可能性がある。
- 記録タブは既に量が増えているため、アイドル由来の読み物、施設由来の記録、廻後追記をどう分類して見失わせないかを別途整理する。

### 廻後清算 UI

- 廻の清算画面で、通常進行へ戻る主導線を1つに見せる最小整理は実装済み。今後は、清算中に迷わせないことを維持しつつ、清算画面と廻タブの役割分担を大きく見直すかどうかを別途検討する。

### 廻後バフ / 再進行速度

- 現状の廻後バフは弱く、廻を実行したい気持ちにつながりにくい可能性がある。次周回の序盤は、初回より明確にサクサク進む体感を目指して、記憶断片獲得量、バフ価格、効果量、廻後基礎加速をまとめて見直す。

### 施設解放と進行バランス

- 次施設の解放条件が全体的に低めで、施設を入手してから次施設解放までにやることが少ない印象になる可能性がある。
- ただしこれは通しプレイ前の仮説なので、まず現行実装済み施設を一通りプレイし、どの区間で待ち時間、強化回数、記録解放、アイドル加入、歌やアイテム購入が足りないかを確認してから調整する。
- 調整する場合は、単に解放レベルを上げるだけでなく、施設ごとの役割、途中の記録解放、歌 / アイテム購入、交流イベントの配置も合わせて見る。

### 検討順の仮置き

1. 廻情報をタブとして残すか、バインダー / 状態確認 / 清算画面などへ分散整理するかを決める。
2. 記録タブの分類と、アイドル詳細側へ移した方がよい読み物の境界を決める。
3. 廻後バフと施設解放バランスは、通しプレイの観察後に数値調整する。
