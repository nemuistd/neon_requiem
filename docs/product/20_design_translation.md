# Design Translation

## 位置づけ

この文書は、[Experience Requirements](10_product_requirements.md) を設計判断へ翻訳するための中間レイヤである。

ここでは、具体的な実装タスクを急いで決めない。UI、文言、コンテンツ、進行設計を考える時に、どのような判断基準を使うかを整理する。

## UI設計への翻訳

UIは、最初に何をすればよいかを明確にしつつ、地下都市の雰囲気を壊さない。

- 初回画面では、最初のアイドル、現在の灯るさ、「ライブする」、最初の復興対象が視線の中心になる。
- 説明文は、操作を止める長文ではなく、状態表示、解放条件、短いメッセージで支える。
- タブやパネルは、施設、歌、アイテム、アイドル、記録の役割を混ぜない。
- 深層語は通常UIの主導線に置かず、記録や後続進行で扱う。
- 放置後の復帰表示は、数値だけでなく「戻ってきた」感覚を支える。

## 文言設計への翻訳

文言は、効率説明と復興感を両立する。

- 「灯るさ」は数値リソース名として使い、演出文では「灯り」も許容する。
- 「聖歌」は通常システム名にしない。通常UIでは「歌」「楽曲」「公演」「演目」など表向きの語を優先する。
- プレイヤーとアイドルの関係は、所有、搾取、管理対象に見えない表現にする。
- メッセージや記録は、暗さよりも回復、居場所づくり、復興を中心に置く。
- 深層文言は、分からない単語を増やすためではなく、過去の体験を再解釈させるために使う。

## コンテンツ設計への翻訳

新しいコンテンツは、数値効果だけでなく「何が戻るのか」を持つ必要がある。

- 施設は、地下都市の復興対象として扱う。
- 歌は、ライブや施設生産に響きを加える恒久アップグレードとして扱う。
- アイテムは、小さな復興活動の手触りを補強する恒久アップグレードとして扱う。
- アイドルは、効率、居場所、進行上の役割、記録や文言との結びつきを持つ。
- 記録は、世界観、主人公の忘れていたこと、区画で得た知識を開示する。

新しいコンテンツを設計する時は、次の問いに答える。

- これは何を増やすのか。
- これはどの場所、誰、どの記憶を戻すのか。
- これはプレイヤーに新しい判断か、気持ちの変化を生むのか。
- これは序盤に出すべき表向きの要素か、後で開示すべき深層要素か。
- これは既存の施設、歌、アイテム、アイドル、記録の役割を濁していないか。

## 進行設計への翻訳

進行は、成長速度だけでなく、意味の開示速度を持つ。

- 序盤は、ライブ、灯るさ、施設強化、自動生産を分かりやすくする。
- 短期では、最初の記録や小さなアップグレードによって「街が少し戻る」感覚を作る。
- 中期では、ネオン掲示板、地下礼拝堂、追加アイドルを通じて、場所と人物の意味を広げる。
- 長期では、霞、観測者、アンカー、祈念工学、聖歌を段階的に扱う。
- 再固定 / 再観測のような仕組みは、実装前に [world_reveal_and_prestige_plan.md](../planning/world_reveal_and_prestige_plan.md) と照合する。

## 技術設計への翻訳

技術判断は、軽さ、保存の安心感、拡張しやすさを優先する。

- 静的 Web アプリとして、バックエンドなしで動作する。
- 保存は `localStorage` を前提にし、壊れたセーブでクラッシュしない。
- ゲームロジック、保存処理、経済計算、コンテンツ定義は TypeScript 側で扱う。
- コンテンツ定義は UI コードへ直書きしすぎず、データとして分離する。
- GitHub Pages に載せやすい相対パス構成を維持する。
- コンテンツ定義を変える場合は、検証方法まで同時に考える。
