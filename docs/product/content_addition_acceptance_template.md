# Content Addition Acceptance Template

## 位置づけ

このテンプレートは、新しい施設、歌、アイテム、アイドル、記録、交流イベントを追加する前に、Product Charter、現在仕様、世界観正史、長期計画とのずれを確認するために使う。

新規コンテンツは、単に数値を増やすものではなく、プレイヤーが「どの場所を戻しているのか」「誰と関わっているのか」「何が少し変わったのか」を感じられる必要がある。

## Product Charterとの対応

- この追加で、プレイヤーは何を楽しめるようになるか。
- 回復、居場所づくり、復興のどれに貢献するか。
- プレイヤーがアイドルを所有・搾取しているように見えないか。
- 深層語や核心設定を急いで開示していないか。

## 体験上の役割

- 初回体験を分かりやすくするものか。
- 中期進行の目標を作るものか。
- 長期進行や再読の意味を作るものか。
- 数値効率以外に、場所、人物、記録、雰囲気の変化があるか。

## 種別ごとの確認

### 施設

- 解放条件が現在の施設進行と矛盾していない。
- どの区画を復興しているかが分かる。
- 生産量、コスト、解放順が急激すぎない。
- `docs/planning/facility_progression_strategy.md` と矛盾しない。

### 歌

- 通常システム名として「聖歌」を使っていない。
- 効果がライブ、公演、演目として自然に読める。
- `docs/worldbuilding.md` の「公演は儀式ではない」「歌は聖歌ではない」に反しない。

### アイテム

- 備品、告知、復興道具として役割が分かる。
- 効果が施設やライブの体験と結びついている。
- 便利さだけでなく、場所が戻る感覚に寄与している。

### アイドル

- 現在実装済みのアイドルだけで物語を閉じていない。
- 追加アイドルが、区画、機能、深層、廻後のどの役割に近いか整理されている。
- 条件達成と加入済み状態を混同していない。
- 全員を全灯るさ生産倍率だけにしていない。
- `docs/planning/idol_roster_expansion_design.md` と矛盾しない。

### 記録

- `revealLevel` が適切である。
- `surface` に祈念工学、アンカー、祈念負荷、聖歌を出していない。
- `technical` / `deep` でも主人公の正体、霞の意志、旧計画の全貌を勝手に確定していない。
- 読んでも灯るさや生産量が変化しない。

### 交流イベント

- `bond` と解放条件の関係が明確である。
- `eventIdsRead` による既読管理が必要か判断している。
- 数値報酬より、アイドルと場所の変化を優先している。
- 既存の記録タブと役割が重複しすぎていない。

## 同期が必要な文書

- 実装済み仕様が変わる場合: `docs/current_spec.md`
- 世界観や用語体系が変わる場合: `docs/worldbuilding.md`
- 長期構想や未実装計画が変わる場合: `docs/planning/`
- Product方針やタスク階層が変わる場合: `docs/product/`
- QA観点が変わる場合: `docs/manual_qa_checklist.md`

## 受け入れ条件

- 目的、対象範囲、受け入れ条件、確認方法が明確である。
- 既存の実装済み仕様と矛盾しない。
- 世界観正史と矛盾しない。
- 必要な同期文書が更新されている。
- 必要な確認コマンドまたは手動確認が実行されている。

## 確認コマンド候補

```sh
npm.cmd run check:content
npm.cmd run test
npm.cmd run build
```

文書のみの場合は、対象ファイルをUTF-8で読み、リンクと参照先を確認する。
