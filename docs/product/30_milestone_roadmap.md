# Milestone Roadmap

## 位置づけ

この文書は、[Design Translation](20_design_translation.md) を、段階的に満たすべき状態へ落としたロードマップである。

ここでは、各マイルストーンが「どんな状態になれば前へ進めるか」、どの順で実行するか、どこまで並列化できるかを整理する。具体的な作業指示は [Task Backlog](40_task_backlog.md) と [Parallel Execution Plan](50_parallel_execution_plan.md) で扱う。

## 実行順の原則

1. M0: 文書階層と実行計画を整える。
2. M1: 初回3分体験を確認できる状態にする。
3. M2: 初期文言と復興感の点検へ進む。
4. M4: 記録の開示レイヤーを点検する。
5. M3: アイドル交流イベントの最小仕様を設計する。
6. M5: 新規コンテンツ追加時の受け入れ条件テンプレートと統合運用を整える。

M1、M2、M3、M4、M5の一部は並列に進められる。ただし、文言トーン、記録の深層語、Product Charterとの整合は最後に統合レビューする。

## M0: 文書階層を整える

目的: Product Charter からタスクバックログまでの階層を作り、下位文書から上位文書へ戻れる運用を明確にする。

実行タスク:

- `P-INTEGRATE-06`: 並列作業後の統合レビュー。

依存関係:

- すべてのタスクの合流後に実行する。

並列化:

- 並列不可。統合専任で実行する。

達成状態:

- `docs/product/` に、`00`、`10`、`20`、`30`、`40`、`50`、`90` の文書がある。
- 各文書の役割が重複しすぎていない。
- 旧タスク直行型の文書構造に依存した参照が残っていない。
- 並列ユニットのwrite scopeと合流順が明文化されている。

未解決論点:

- 既存の `docs/planning/` と `docs/product/` の役割境界を、今後の運用で必要に応じて磨く。

## M1: 初回3分体験を固める

目的: 初回プレイで、ライブ、灯るさ、施設強化の関係を迷わず理解できる状態にする。

実行タスク:

- `P-M1-QA-01`: 初回3分導線QAを既存チェックリストへ落とす。
- `P-M1-COPY-02`: 初期メッセージ・灯るさ説明・主要フィードバック文言を点検する。

依存関係:

- `P-M1-QA-01` は他タスクに依存しない。
- `P-M1-COPY-02` は `P-M1-QA-01` と並列に着手できるが、QA観点の最終反映は統合レビューで確認する。

並列化:

- `P-M1-QA-01` はQA文書のみを触るため、他タスクと並列可。
- `P-M1-COPY-02` は `src/data.ts` を中心に触るため、記録定義を触る `P-M4-RECORD-03` と並列可。

達成状態:

- 新規セーブで、最初に押すべき操作が分かる。
- 灯るさの意味が、単なる通貨ではなく復興と結びついている。
- 最初の施設強化までの導線が、説明を読ませすぎずに通る。
- 深層語を理解しなくても序盤の目標が分かる。

未解決論点:

- 初回3分の実ブラウザQAを、既存チェックリストのどちらに統合するか。
- 初期メッセージの文言が、現状のUI密度で十分に伝わるか。

## M2: 成長と復興感を結びつける

目的: 数値成長が、街や場所が戻る感情として受け取れる状態にする。

実行タスク:

- `P-M1-COPY-02`: 初期メッセージ・灯るさ説明・主要フィードバック文言を点検する。
- `P-M4-RECORD-03`: 既存記録の `revealLevel` と深層語の出し分けを点検する。

依存関係:

- どちらも単独で開始できる。
- 最終的な文言トーンは `P-INTEGRATE-06` で確認する。

並列化:

- 並列可。`P-M1-COPY-02` はUI文言、`P-M4-RECORD-03` は記録本文と開示レイヤーが中心で、write scopeが分かれる。

達成状態:

- 施設、歌、アイテムが、それぞれ何を戻しているのか読める。
- 強化や購入時のフィードバックが、役割に合っている。
- 最初の記録が、世界の見え方を少し変える報酬として機能している。
- コンテンツ定義と現在仕様が食い違っていない。

未解決論点:

- 既存の説明文だけで復興感が十分か。
- 記録解放タイミングの調整が必要か。

## M3: アイドルを個人として感じられる導線を作る

目的: アイドルを倍率装置ではなく、支える対象であり居場所を持つ個人として感じられる状態にする。

実行タスク:

- `P-M3-IDOL-04`: アイドル交流イベントの最小仕様を設計文書化する。

依存関係:

- 実装変更を伴わないため、M1/M2/M4の点検と並列に進められる。
- `bond` 実装済み仕様と `docs/planning/idol_roster_expansion_design.md` を確認してから設計する。

並列化:

- 並列可。write scopeはplanning docs中心。

達成状態:

- 注目アイドル枠とアイドルタブの文言が、所有や管理対象に見えない。
- `bond` の「交流」としての意味が、ライブ時の行動と結びついている。
- アイドル解放が、効率上昇だけでなく、新しい区画や記録への入口として機能している。
- 将来の交流イベントを設計する前提が整理されている。

未解決論点:

- 交流イベントをどのタイミングで最小実装するか。
- イベントを数値報酬に寄せすぎないための型や表示設計が必要か。

## M4: 深層開示と長期進行を整える

目的: 世界観の謎を、説明量ではなく再解釈で深められる状態にする。

実行タスク:

- `P-M4-RECORD-03`: 既存記録の `revealLevel` と深層語の出し分けを点検する。

依存関係:

- `P-M1-COPY-02` と並列に進められる。
- `worldbuilding.md` と `contentValidation.ts` の深層語ルールを確認してから判断する。

並列化:

- 並列可。ただし、コンテンツ定義を変更した場合は合流後に `npm.cmd run check:content` を再実行する。

達成状態:

- 記録の `surface`、`uncanny`、`technical`、`deep` の出し分けが整理されている。
- 序盤では深層語を出しすぎず、中盤以降に異常の気配が増える。
- 主人公の正体、霞の意志、旧計画の全貌を勝手に確定していない。
- 再固定 / 再観測の候補は、実装前の計画段階に留まっている。

未解決論点:

- どの進行段階から technical / deep の記録を増やすか。
- 再固定 / 再観測を実装する前に、どの施設や記録を追加すべきか。

## M5: タスクバックログを運用する

目的: 十分に具体化した項目だけを、順序付きタスクとして実装に渡せる状態にする。

実行タスク:

- `P-M5-TEMPLATE-05`: 新規コンテンツ追加時の受け入れ条件テンプレートを作る。
- `P-INTEGRATE-06`: 並列作業後の統合レビュー。

依存関係:

- `P-M5-TEMPLATE-05` はM1/M3/M4と並列に進められる。
- `P-INTEGRATE-06` はすべての作業ユニット完了後に行う。

並列化:

- `P-M5-TEMPLATE-05` はproduct docsのみを触るため、実装・コンテンツ点検系タスクと並列可。
- `P-INTEGRATE-06` は並列不可。

達成状態:

- 各タスクに、目的、対象範囲、受け入れ条件、確認方法がある。
- タスクが Product Charter と Experience Requirements に紐づいている。
- 実装中に上位文書の改善が必要になった場合の戻り先が明確である。

未解決論点:

- どのマイルストーンを次の実装対象にするか。
- 実ブラウザQAをどの粒度でタスクへ含めるか。
