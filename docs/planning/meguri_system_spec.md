# 廻本体 仕様メモ

## 位置づけ

この文書は、廻本体の基本仕様と今後の拡張判断を整理する。

正史・物語面は `docs/fiction/world_design.md` と `docs/fiction/meguri_after_design.md` を優先する。この文書は、それらを実装タスクへ落とすための仕様メモであり、実装済みの内容と将来調整対象を併記する。古い planning 側の未定扱いや仮配置が fiction と食い違う場合は、fiction を優先する。

現在の実装には、再固定中枢到達、任意の廻実行、記憶断片清算、次断片ゲージ、廻後の専用清算画面、廻後ダッシュボード、追記索引、初期4件の記録追記表示、追記中心の記録通知が入っている。fiction 正史上の Ch.7 以降施設、霞山 澪、七城 皐月、白霧 燐、Ch.9 収束は未実装。

## 基本方針

- 廻前のコンテンツは、現在の進行順に沿って一通り解放される。
- プレイヤーが廻前の最後の導線まで到達すると、自然に `再固定中枢` へ至る。
- 廻は強制ではなく、プレイヤーが任意に選ぶ。
- 廻の実行は大きな状態変更なので、実行前にリセット対象と引き継ぎ対象を明示する。
- 廻は「転生特典だけを受け取る操作」ではなく、「街を一度たたみ、痕跡を次へ持ち越す操作」として扱う。

## 発動条件

初回廻の発動条件は、廻前コンテンツを順番に進めた結果として満たされる形にする。

v1 実装済み条件:

- 地下通路修復区画 Lv5。
- `修復の仮歌` 取得。
- `再固定中枢` Lv1。

将来調整候補:

- 地下通路修復区画の終盤レベルに到達している。
- 廻前の主要記録・歌・アイテム・アイドル解放の導線が一通り完了している。

廻が実行可能になった後は、実行前の確認画面で「今ここで廻ると獲得できる記憶断片の見込み」を表示する。これは実際の付与処理と同じ算出式によるプレビューであり、廻を実行するまでは記憶断片を付与しない。

## リセット対象

廻を実行すると、通常進行で積み上げたものはほぼリセットする。

- 灯るさの現在値
- 施設レベル
- 歌の購入状態
- アイテムの購入状態
- アイドルの bond 数値
- 廻前の一時的な UI 状態

注目アイドルの選択は、保存実装上はリセットしても保持してもよい。ただし、選択中のアイドルが再解放前の場合は、表示上は未解放扱いに戻す。

## 引き継ぎ対象

プレイヤーの負担を減らし、世界に痕跡が残る感覚を出すため、記録まわりは強く保持する。

- 解放済み記録
- 既読記録
- 廻後アノテーションの解放状態
- 廻回数
- 記憶断片
- 購入済み廻後バフ
- アイドルの認識フラグ
- バフ購入に紐づく追記表示フラグ

アイドルの認識フラグは、bond 数値そのものではなく、「以前の廻で一定以上関わった」という痕跡として保持する。これにより、bond はリセットされても、薄明の記憶や微細なセリフ変化へ接続できる。

## 記憶断片

記憶断片は、廻を実行した時に獲得する引き継ぎ可能な資源とする。

基本方針:

- 記憶断片は通常プレイ中に常時生産しない。
- 記憶断片は、全周回における灯るさ獲得総量に応じて算出する。
- 廻実行時に、総量から算出される累計獲得可能量と、これまでに獲得済みの記憶断片累計との差分を付与する。
- これにより、同じ総量から何度も記憶断片を受け取ることを防ぐ。
- 記憶断片は廻をまたいで保持する。
- 消費済み記憶断片は戻さない。
- 廻前の確認画面では、同じ計算に基づく獲得見込みを表示する。
- 獲得見込みには、購入済みの記憶断片獲得量バフを反映する。
- まだ購入していない候補バフ、または廻後の清算画面でこれから購入するバフは、今回の獲得見込みには含めない。

実装時の注意:

- プレビュー表示と実際の付与処理は、同じ純粋関数を使う。
- 「購入済みバフを反映した獲得見込み」と「廻実行時に実際に付与された記憶断片」が一致するテストを入れる。
- UI 表示用に丸めた値と、保存・付与に使う値の丸め位置を揃える。
- v1 では、全周回累計の灯るさから算出した累計獲得可能量に、購入済みの記憶断片獲得量バフを反映する。

実装時には、以下のような保存値が必要になる可能性がある。

- 全周回累計の灯るさ獲得量
- 記憶断片の現在値
- 記憶断片の累計獲得量

v1 実装済みの算出式:

```ts
floor(sqrt(totalTomorusaEarned / 20000) * memoryFragmentMultiplier)
```

今回付与量は、上記の累計獲得可能量から `totalMemoryFragmentsEarned` を差し引いた値とする。

灯るさの指数的な伸びをそのまま反映すると廻後バフが急激に膨らむため、今後も算出式を変更する場合は、対数、平方根、段階テーブルなど、伸びを圧縮する式を優先する。

## 記憶断片プレビューと次断片ゲージ

廻前の確認画面では、「今廻ると何個もらえるか」だけでなく、「あと少し進めると次の記憶断片に届くか」を判断できるようにする。

表示方針:

- 現在の累計灯るさに基づく獲得見込みを表示する。
- 次の記憶断片に必要な累計灯るさを表示する。
- 「あと X 灯るさで +1 記憶断片」のような短い補助表示を置く。
- 現在値から次の断片までの進捗をゲージで表示する。
- 次の断片が遠すぎる場合は、無理に居座らせず「次の廻で進めてもよい」と読める表現にする。

v1 の算出式を使う場合、次の記憶断片に必要な累計灯るさは以下から逆算できる。

```ts
ceil((nextEligibleFragments / memoryFragmentMultiplier) ** 2 * 20000)
```

注意:

- ゲージ表示、獲得見込み、実際の付与処理は、同じ計算関数群から値を得る。
- ゲージのために別の丸めや近似式を持たない。
- 記憶断片獲得量バフを購入済みの場合、ゲージにも同じ倍率を反映する。
- 初回廻前でも、再固定中枢に到達しているなら獲得見込みとゲージは見せる。
- 記憶断片が 0 個のまま廻れる場合は、それが分かるように表示する。

## 廻後バフ

廻後バフは、記憶断片を消費して購入する。

購入タイミング:

- バフを購入できるのは、廻実行直後の清算画面、または廻後の専用バフ購入フェーズに限定する。
- 通常プレイ中は、購入済みバフと候補バフの確認だけ可能にする。
- 通常プレイ中に施設や灯るさを維持したままバフだけ購入できる状態にはしない。

画面方針:

- バフ購入は専用画面で行う。
- 既存の施設、歌、アイテム、記録、アイドル画面へ無理に詰め込まない。
- 通常プレイ中でも、購入済みバフの一覧と次回廻時に購入できそうな候補は確認できるようにする。
- 廻実行前の確認画面では、現在の進行で廻った場合に獲得できる記憶断片の見込みを表示する。
- この見込みは、購入済みの記憶断片獲得量バフを含めた最終見込みとして表示する。
- 廻実行直後は、獲得した記憶断片、購入可能バフ、購入済みバフをまとめて確認できるようにする。

候補カテゴリ:

- 灯るさ生産倍率
- ライブ獲得量
- オフライン報酬
- bond 上昇速度
- 記録アノテーション解放
- 記憶断片獲得効率

### バフ量の調整方針

初期バフは、初回廻で少なくとも1つ購入できる可能性を持たせるため、v1 初期値から価格を下げ、効果を少しだけ上げている。プレイヤーが「もう少し稼いでから廻るか」「今すぐ次へ行くか」を判断できる UI が整っても、数値だけを大きくしすぎない。

現在値:

- `足元に残る残響`: 3 記憶断片 / 廻後の施設生産倍率 x1.08。
- `残された作業メモ`: 3 記憶断片 / オフライン灯るさ報酬 x1.15。
- `覚えのある呼び名`: 4 記憶断片 / 交流増加量 x1.15。
- `断片の索引`: 5 記憶断片 / 記憶断片の獲得見込み +20%。

次の調整候補:

- 実測を見て、初回廻で買える候補数を調整する。
- 効果量を上げる場合でも、好きなアイドルや寄り道を選べない最適解に寄せすぎない。

## アノテーション

廻後アノテーションは、バフ購入と連動して解放する。

- 記録の追記を直接コストで購入しない。
- バフを購入すると、そのテーマに関連する記録の追記が表示される。
- 解放済み記録に追記が追加された場合は、記録一覧で新しい追記があることを示す。
- 既読状態は維持するが、追記が増えた記録は再読する意味が分かる表示にする。

この方針により、「強化を買う」と「世界の読み方が変わる」が同時に起きる。

## 記録通知ポリシー

廻後の記録通知は、「廻ったから既読をもう一度全部読ませる」ためのものではなく、新しい本文、追記、または表示状態の変化を知らせるためのものに限定する。

通知を出す条件:

- 新しい記録が解放された。
- 既存記録に新しい追記が追加された。
- 記録本文または追記の表示バージョンが変わった。
- プレイヤーがまだ読んでいない追記がある。

通知を出さない条件:

- 廻によって、解放済み記録がそのまま引き継がれただけ。
- 既読済み本文が、内容変更なしで再表示されただけ。
- 清算画面へ入っただけで、記録内容に変化がない。

実装方針:

- 既読状態と追記既読状態を分ける。
- 記録一覧の未読バッジは、本文未読と追記未読を区別できる形にする。
- 廻実行時に `recordTabLastSeenContentVersion` だけを巻き戻して、不要な通知が出る状態にしない。
- 将来の本文更新に備え、必要なら `bodyVersion` と `annotationVersion`、またはそれに相当する通知キーを持たせる。

## 再固定中枢

`再固定中枢` は、実装上は通常施設に近い形で扱う。

役割:

- 廻前最後の導線として到達する施設。
- 廻選択肢を開く物語上の鍵。
- 灯るさを生産する通常施設。
- 記憶断片を常時生産する施設ではない。

世界観上の設定:

`再固定中枢` は、復興した街の状態を一時的に固定し、次の廻へ持ち越せる痕跡へ圧縮する場所である。施設が稼働している間に記憶断片を少しずつ作るのではなく、廻を選んだ瞬間に、これまで積み上げた灯るさ、記録、名前、観測の痕跡を記憶断片へ変換する。

この扱いにすると、再固定中枢はゲーム経済上は扱いやすい通常施設でありながら、物語上は廻の意味を支える中心施設になる。

## 清算フェーズ UI

清算中は、通常プレイ中と見た目も操作も明確に分ける。単に廻タブ内で購入可能状態にするだけだと、「何をしたら清算が終わるのか」「今だけ買えるのか」が分かりにくい。

推奨方針:

- 廻実行後は、専用の清算画面へ遷移する。
- 清算画面では、獲得した記憶断片、購入可能バフ、購入済みバフ、追記が増えた記録だけを中心に表示する。
- 通常のライブ、施設強化、歌取得、アイテム購入へ視線が移りすぎないようにする。
- `pendingSettlement === true` の間は、画面上に「清算中」であることを明確に出す。
- 清算を終える操作は、専用の「通常進行へ戻る」ボタンを第一候補にする。
- 追記が増えた記録がある場合は、対象記録の控えを清算画面に出し、主ボタンとは別の控えめな「記録タブで追記を読む」導線で清算終了と記録確認をつなげる。
- 記録タブへ移る導線でも清算フェーズが終わることを、導線の近くで明示する。
- 通常進行の操作で清算が閉じる仕様にする場合も、その操作で清算フェーズが終わることを事前に分かるようにする。

保存・復元:

- リロード後も `pendingSettlement === true` なら清算画面へ戻す。
- 清算画面から通常進行へ戻った後は、バフ購入は確認のみとし、追加購入できない。
- 清算終了後に記録タブへ通知を出す場合は、記録通知ポリシーに沿って追記や本文更新がある場合に限る。

## 廻後再進行の軽量設計

### 目的

初回廻後に、同じ路地裏から再開する意味を少し増やす。ただし、fiction 正史上の Ch.7 以降施設、霞山 澪、七城 皐月、白霧 燐、Ch.9 収束には入らない。

廻後 v3 の狙いは、次の3つに絞る。

1. プレイヤーが「いまは何廻目で、何を持ち越しているか」を確認できる。
2. 購入済み廻後バフと記録追記が、再進行中も埋もれない。
3. アイドルの「以前の廻の痕跡」を、本人が廻を理解している描写ではなく、次の小さな交流へ接続できる。

### 追加するもの

#### 1. 廻後ダッシュボード

廻タブに、通常進行中でも確認できる小さなダッシュボードを置く。

最小実装済み。初回廻後の通常進行中にだけ表示し、新しい保存フィールドは追加しない。

表示する情報:

- 第何廻か。
- 現在の記憶断片。
- 購入済み廻後バフ数。
- 追記が解放されている記録数。
- 「以前の廻の痕跡」が残っているアイドル数。
- 次に見える小さな目標。

このダッシュボードは、数値を増やす新システムではない。既存の `meguri.count`, `resources.memoryFragment`, `meguri.buffs`, `records`, `idolRecognition` を見やすくまとめるだけにする。

#### 2. 追記索引

廻タブに、廻後バフで追記が解放された記録の索引を置く。

最小実装済み。初回廻後の通常進行中に表示し、関連バフ名は既存の `annotationRequirement` の `meguri.buff.purchased` から逆引きする。追記本文は記録タブに残し、廻タブでは索引と導線だけを持つ。

表示する情報:

- 追記が解放された記録名。
- 追記が既読か未読か。
- 関連する廻後バフ名。

目的は、バフ購入と世界の再読がつながっていることを見失わせないこと。記録本文は記録タブで読む。廻タブ側は索引と導線に留める。

#### 3. 薄明の記憶イベントの入口

廻後に同じアイドルと再び交流を積むと、通常交流イベントとは別に「薄明の記憶」イベントを出せるようにする。

ただし初期実装では、次の制約を置く。

- アイドル本人が廻を理解しているとは書かない。
- 「覚えている」と断定しない。
- 「夢で見た気がする」「この距離を知っている気がする」程度に留める。
- 白霧 燐、霞山 澪、七城 皐月など Ch.7 以降の人物を出さない。

P-M7-IDOL-15 での結論:

- `idolRecognition` は薄明の記憶イベント専用の隠し判定ではなく、`Requirement` に `meguri.idolRecognition` として追加する。
- `meguri.idolRecognition` は `state.meguri.idolRecognition[idolId] === true` を見るだけで、アイドル本人が記憶を保持していることを意味しない。
- `IdolEventDefinition` には `eventKind?: "normal" | "twilightMemory"` を追加し、省略時は `"normal"` と扱う。
- 既読状態は既存の `eventIdsRead` を使う。薄明の記憶イベント専用の保存フィールドは追加しない。
- 表示では通常交流イベントと同じカード基盤を使いつつ、見出しまたは小さなラベルで「薄明の記憶」と分ける。既存イベント本文は差し替えない。
- 最初の実装対象は音羽 灯里 1件に絞る。ID 候補は `otowaAkari.twilightFirstPause`。
- 解放条件は `all([meguri.count >= 1, meguri.idolRecognition(otowaAkari), idol.bond(otowaAkari) >= 5])` とする。
- 本文方針は、灯里の第一声や返事の間合いが少し違う程度に留める。「前にも会った」「廻を覚えている」とは書かない。

### まだ追加しないもの

- Ch.7以降の新アイドル（霞山 澪、七城 皐月、白霧 燐）。
- 白霧 燐の名前、姿、台詞。
- Ch.9収束。
- バインダー封書の全文解読。
- 主人公の正体、霞の意志、旧計画の全貌の確定。
- 施設レベルの持ち越し。
- 廻後に通常進行中でもバフを追加購入できる仕組み。
- 記憶断片の常時生産。
- 複数追記を1つの記録へ重ねるための大きな記録モデル変更。
