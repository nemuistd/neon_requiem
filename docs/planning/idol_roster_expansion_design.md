# アイドル拡張設計

## 位置づけ

この文書は、アイドルを将来的に10人以上へ増やすことを前提に、現在と今後の設計方針を整理する。
承認済みの人物像・章配置・解放条件は、後から作られた `docs/fiction/idol_profiles.md` と `docs/fiction/content_design.md` を優先する。この文書の古い仮ロスターや役割候補が fiction と食い違う場合は、fiction を正として読み替える。

## 大前提

現在実装済みのアイドルは、物語全体を完結させるための全メンバーではない。
承認済み fiction では当面の主要ロスターとして10人が定義されている。実装済みメンバーはその一部であり、プロローグ組、区画担当組、深層接続担当として扱い、後続メンバーの余白を残す。

## 主要プロローグ組の位置づけ

### 音羽 灯里

- 初期アイドル。
- 路地裏ステージ、最初のライブ、最初の灯るさに関わる。
- プレイヤーに基本ループを教える。
- 世界観核心の解決者ではなく、最初の焦点。

### 朝霧 結

- ネオン掲示板、薄明通り、道案内、霞の濃い通路への導線。
- 序盤から中盤へ進む橋渡し。
- 地下礼拝堂より前に、霞や道の不安定さを自然に見せる役割を担える。

### 深月 詩乃

- 地下礼拝堂、失われた響き、保管された記録への導線。
- 深層要素に接続しやすいが、序盤で聖歌や祈念工学を解決する役ではない。
- 肩書きや説明は、通常UIで「聖歌係」に戻さない。

## 今後のアイドル分類

### 1. プロローグ組

灯里・結・詩乃を中心とするプロローグ組。
世界観と基本ループの入口を作る。

プロローグ組の役割は、ゲームの最初の理解を支えることである。
この組だけで、霞、旧計画、主人公の正体、聖歌の全貌を解決しない。

現在は、響木 遠子、紙野 巡、陽向 小春、継ノ端 桜子も最小実装済みであり、区画担当組や深層接続担当の入口として扱う。

### 2. 区画担当組

追加アイドルの主力。
各アイドルが新しい施設群・区画・プレイ要素に対応する。

候補例:

- 配信ブース系アイドル
- 記憶図書館系アイドル
- 地下駅跡 / 移動導線系アイドル
- 市場 / 広場系アイドル
- 修復工房系アイドル
- 記録保管区系アイドル
- ファンコミュニティ系アイドル

### 3. 機能担当組

プレイ要素の幅を広げるためのアイドル。
区画担当組と重なってよいが、効果設計では役割を明確にする。

候補例:

- ライブ獲得を伸ばす担当
- 特定施設群を伸ばす担当
- オフライン報酬を伸ばす担当
- アイテムや歌の取得を補助する担当
- 記録解放や既読導線を補助する担当
- 将来のファン資源や交流値を伸ばす担当

### 4. 深層担当組

中盤後半から後半に登場。
祈念工学、アンカー、記憶封印、旧計画などへ接続する。

深層担当組は、表向きの復興ゲームを壊さない範囲で断片を扱う。
一人のアイドルだけに世界観核心を背負わせない。

### 5. 廻後担当組

廻後に意味が見えるアイドル。
1周目では名前や記録だけが見え、2周目以降に役割が変わって見える形も許容する。

fiction 正史では、白霧 燐が Ch.9 / 廻2の廻後限定アイドルとして定義されている。廻1では記録やバインダーに名前だけが影として出る。燐は「廻の記憶を持つように見える」が、証人や答えを持つ説明役としては扱わない。
「1周目ではまだ意味が分からないが、後で読み替えられる」余白を残す。

## アイドル能力設計

10人以上になるため、全員を単純な「全灯るさ生産倍率」にしない。
全体倍率は分かりやすいが、人数が増えるほど効果が単調になり、注目アイドル選択の意味も薄くなる。

推奨する効果カテゴリ:

- 全灯るさ生産倍率
- ライブ獲得量
- オフライン報酬倍率
- 特定施設生産倍率
- アイテム価格軽減
- 歌 / アイテム解放補助
- 記録解放補助
- ファン獲得量
- 交流値獲得量
- 廻後ボーナス

効果設計の目安:

- プロローグ組は、基本ループの理解を助ける単純な効果でよい。
- 区画担当組は、対応する施設や区画に効果を寄せる。
- 機能担当組は、ライブ、施設、オフライン、記録、アイテムなどの軸を分担する。
- 深層担当組は、強力な数値効果よりも開示や条件補助を優先する。
- 廻後担当組は、1周目から必須にしない。

## アイドル加入と効果解放の方針

アイドルは、条件を満たした瞬間に自動で効果が発動するだけだと、プレイヤーが「出会った」「声をかけた」「一緒に場所を戻し始めた」という実感を得にくい。現在の実装では、解放条件を満たした状態と、実際に加入して効果が発動する状態を分けている。

状態の考え方:

- 未発見: 条件を満たしておらず、通常UIには出さない。
- 出会い可能: 解放条件を満たしたが、まだ加入していない。
- 加入済み: プレイヤーが能動的な操作を行い、プロデュース対象として加わった。
- 再解放待ち: 廻後に通常進行がリセットされ、条件を満たすまで表示や交流が制限されている。

UI 方針:

- 条件を満たしたら、アイドルタブや関連区画に「声をかける」「プロデュースする」「迎える」などのアクションを出す。
- 文言は、所有、購入、雇用に見えないものを優先する。
- 加入操作の直後に、そのアイドルの効果が解放されたことを短く表示する。
- 効果表示は、数値だけでなく「結が道案内を手伝ってくれるようになった」のように、場所や役割と結びつける。
- 初期アイドルの音羽 灯里は、新規ゲーム開始時点で加入済みとして扱う。

効果適用:

- `passiveEffects` は、条件達成済みではなく加入済みアイドルだけを集計する。
- 注目アイドルに選べるのも、原則として加入済みアイドルに限る。
- 条件達成済みだが未加入のアイドルは、効果を発動しない。
- ただし、出会い可能状態のカードや短い導入文は表示してよい。

保存・移行:

- 実装では `GameState.idols[id].joined`、または同等の `recruited` フラグを追加する。
- 既存セーブでは、すでに解放済みだったアイドルを加入済みに移行し、プレイヤーの既存バフを突然失わせない。
- 廻実行時に加入状態をリセットするかどうかは、通常進行の解放状態と廻後の認識フラグの役割を分けて決める。v1 の方針では、廻前の通常進行としての加入状態はリセット候補、`idolRecognition` は痕跡として保持する。
- 加入状態をリセットする場合でも、認識フラグがあるアイドルには「以前の廻の痕跡」程度の薄い表示差分を許容する。

実装済み:

- `isIdolUnlocked` と `isIdolJoined` を分ける。
- アイドル効果集計を、加入済みアイドルだけに変更する。
- 条件達成済みだが未加入のカード表示と加入ボタンを追加する。
- 加入時メッセージと効果解放表示を追加する。
- 既存セーブの解放済みアイドルを加入済みへ移行する。
- 未加入では効果が発動しないこと、加入後に発動すること、移行で既存効果が消えないことをテストする。

## 薄明の記憶イベント

薄明の記憶イベントは、廻後に「以前の廻の痕跡」が残っているアイドルと、もう一度小さく交流した時に出る短い読み物として扱う。

このイベントは、アイドルが廻を理解している、または前の廻を明確に覚えていることを示すものではない。本人の言葉は「この距離を知っている気がする」「初めてなのに少し安心する」程度に留め、意味が分かるのは前の廻を知っているプレイヤー側だけにする。

実装方針:

- `idolRecognition` は、本人の記憶ではなく「以前の廻で一定以上関わった痕跡」として扱う。
- 実装では `Requirement` に `meguri.idolRecognition` を追加し、通常の条件体系から参照できるようにする。
- `IdolEventDefinition` に `eventKind?: "normal" | "twilightMemory"` を追加し、通常交流イベントと薄明の記憶イベントを分類する。
- 既読状態は既存の `eventIdsRead` を使い、薄明の記憶イベント専用の保存フィールドは追加しない。
- 表示場所はアイドルタブと左側の注目アイドル詳細パネルを優先する。記録タブには混ぜない。

最初の対象:

| アイドル | イベントID候補 | 条件 | 強度 | 方針 |
| --- | --- | --- | --- | --- |
| 音羽 灯里 | `otowaAkari.twilightFirstPause` | `meguri.count >= 1`、灯里の `idolRecognition`、灯里の再交流 `bond >= 5` | 弱 | 第一声や返事の間合いが少し違う。「覚えている」とは言わない。 |

次以降の候補:

- 響木 遠子は「声を届けたことがある気がする」程度の弱いイベント候補にできる。
- 深月 詩乃は fiction 上は中程度の記憶強度候補だが、地下礼拝堂や記録保管の意味が強いため、初回最小実装には含めない。
- 朝霧 結と陽向 小春は、現時点では薄明の記憶イベントを無理に追加しない。fiction 上の「変化なし」または通常の再会感を尊重する。
- 霞山 澪、七城 皐月、白霧 燐は Ch.7 以降の正史範囲なので、今回の薄明の記憶イベント最小実装には含めない。

## `passiveEffects` と `focusEffects`

現在の実装では、アイドル定義は以下の形を前提にしている。

```ts
type IdolDefinition = {
  ...
  passiveEffects: Effect[];
  focusEffects?: Effect[];
}
```

- `passiveEffects`: 加入済みなら常時発動。
- `focusEffects`: 注目アイドルのときだけ参照できる拡張枠。ただし、進行効率差を作る効果として使うことは原則保留する。

注目アイドルは、まず「好きなアイドルを画面に置く」「一緒にライブして交流を積み重ねる」ための枠である。
ここに強い数値効果を付けると、好きなアイドルを自由に置くほど損をする状況が生まれる。
そのため、`focusEffects` は型や検証上の余地として残しても、実際の効果適用は慎重に保留する。

実装済み方針:

1. 旧 `passiveEffect` と同じ挙動を `passiveEffects` 配列へ移行済み。
2. 実装済みアイドルの効果量や解放条件は、移行で変えない。
3. `focusEffects` は、型として残してよいが、効率上の有利不利を作る用途では使わない。
4. 注目アイドルに追加の意味を持たせる場合は、専用セリフ、演出、交流イベントの見つけやすさなど、好きなアイドルを置く自由を壊さない方向を優先する。
5. contentValidation で Effect 内の参照IDを検証する。

最初の実装時は大きなDSLを作らず、既存の Effect 型を拡張して扱う。

## アイドル効果モデル詳細設計

### 旧モデルの限界

旧アイドル定義は、単一の `passiveEffect` を持っていた。
現在は `passiveEffects` 配列と加入済み判定を使い、実装済みアイドルの効果を集計する。

旧モデルは最初の3人では分かりやすいが、10人以上に増えると以下の問題が出る。

- 全員が全体生産倍率になると、効果差が単調になる。
- 注目アイドルの意味を、効率ではなく表示、交流、個別体験として強める必要がある。
- 特定施設、ライブ、オフライン報酬、記録解放などの役割分担がしにくい。
- 追加アイドルの個性を数値効果に反映しにくい。
- 後から `bond` や廻後効果を接続するとき、単数効果のままでは拡張しづらい。

### `passiveEffects: Effect[]`

`passiveEffects` は、アイドルが加入済みなら常時発動する効果配列とする。
旧 `passiveEffect` は、同じ挙動の1要素配列へ移行済み。

初回移行のイメージ:

```ts
type IdolDefinition = {
  ...
  passiveDescription: string;
  passiveEffects: Effect[];
  focusEffects?: Effect[];
}
```

初期3人の移行結果:

- 音羽 灯里: 現在の全灯るさ生産 `x1.20` を維持する。
- 朝霧 結: 現在の全灯るさ生産 `x1.15` を維持する。
- 深月 詩乃: 現在の全灯るさ生産 `x1.10` を維持する。
- 解放条件、表示文、プレイ感は変えない。

### `focusEffects?: Effect[]`

`focusEffects` は、注目アイドルに選ばれている間だけ参照できる効果候補である。
ただし、注目アイドルはプレイヤーが好きなアイドルを画面に置くための枠でもある。
そのため、進行効率に直結する `focusEffects` は、好きなアイドルを選ぶ自由を損なうリスクが高い。

当面の扱い:

- 型は残してよいが、実効果の適用は保留する。
- 注目中だけライブ獲得量を増やす効果は原則入れない。
- 注目中だけ特定施設の生産を伸ばす効果は原則入れない。
- 注目中だけ `bond` 獲得量を増やす効果は慎重に保留する。
- 注目中だけ価格軽減、記録解放補助、アイテム解放補助を行う効果は原則入れない。
- 注目中の変化は、専用セリフ、表情、メッセージ、交流イベント導線など、効率差を生まない体験差を優先する。

将来どうしても数値効果を導入する場合の条件:

- 好きなアイドルを置くと損をする、と感じさせない。
- 特定アイドル固定が明確な最適解にならない。
- 頻繁な切り替えを要求しない。
- 効果量は極小、または全アイドル間で実質的な差が出ない。
- 効率目的ではなく、交流や演出の補助として説明できる。

### 効果カテゴリ

10人以上に増やす場合、以下のカテゴリから役割を分散する。

| カテゴリ | 常時効果向き | 注目枠での扱い | 備考 |
| --- | --- | --- | --- |
| 全灯るさ生産倍率 | ○ | 原則不可 | 好きなアイドルを置く自由を損ないやすい |
| ライブ獲得量 | ○ | 原則不可 | 効率目的の注目固定を生みやすい |
| 特定施設生産倍率 | ○ | 原則不可 | 区画担当組の個性は常時効果側で扱う |
| オフライン報酬倍率 | ○ | 原則不可 | 不在時の最適化圧になりやすい |
| アイテム価格軽減 | ○ | 原則不可 | 切り替え要求が強くなりやすい |
| 歌 / アイテム解放補助 | ○ | 原則不可 | 好きなアイドルより効率条件を優先させやすい |
| 記録解放補助 | ○ | 原則不可 | 深層開示の順番にも影響する |
| ファン獲得量 | ○ | 原則不可 | ファン資源導入後も常時効果側で検討する |
| bond 獲得量 | ○ | 慎重に保留 | 交流対象の選択以上の効率差を作らない |
| 廻後ボーナス | ○ | 原則不可 | 1周目必須にしない |

### 実装履歴と今後の分割案

1. 型移行は完了済み。
   `IdolDefinition.passiveEffect` を `passiveEffects: Effect[]` へ変更し、既存効果を同じ数値で配列化した。
2. 集計関数は加入済みアイドルの `passiveEffects` を集約する形へ変更済み。
3. contentValidation は Effect 参照IDや数値妥当性を検証対象に含める。
4. UI表示は `passiveDescription` を維持する。
   効果配列から表示文を自動生成しない。
5. `focusEffects?: Effect[]` は効率上の実効果として使わない。
6. 注目アイドルの意味を増やす場合は、まず交流イベント、専用メッセージ、プロフィール表示など、効率差を生まない実装を優先する。

### テスト方針

アイドル効果や加入仕様を変更するPRでは、少なくとも以下を確認する。

- 実装済みアイドルの生産倍率が意図せず変わらない。
- 未解放アイドルの `passiveEffects` は発動しない。
- 出会い可能だが未加入のアイドルの `passiveEffects` は発動しない。
- 加入済みアイドルの複数効果がすべて集計される。
- `focusEffects` を型として持っても、現時点では進行効率に適用されない。
- `contentValidation` が不正な Effect 数値や参照IDを検出する。

## 交流値

実装名は `bond` を第一候補にする。
UI名は「交流」または「親しさ」とする。

導入順:

1. `GameState.idols[id].bond` のような個別状態を追加する。
2. 注目アイドルでライブしたときだけ、少量の bond を増やす。
3. UIに「交流」または「親しさ」として表示する。
4. `idol.bond` Requirement を追加し、記録や小さなイベント解放に使う。
5. その後、交流イベントや専用メッセージとの接続を検討する。`focusEffects` を数値効率へ接続することは保留する。

注意:

- 「信頼」は、将来の人間関係表現として使う余地はあるが、灯し手本人の自己同一性維持資源にはしない。
- 「個性」は重要概念だが、序盤から本格資源化しない。
- 最初はbondを、注目アイドルとの交流値として扱う。

## 交流イベントの最小仕様

交流イベントは、`bond` が増えたことを単なる数値効率ではなく、「一緒に場所を戻している実感」として受け取れるようにするための小さな体験である。

最小実装では、会話イベントを大きなストーリーシステムにしない。既存の記録タブとは役割を分け、アイドル本人との短い変化や反応を扱う。

### 役割

| 役割 | 内容 | 最小実装での扱い |
| --- | --- | --- |
| 効率 | ライブや施設復興に少し関わる | 初回実装では数値報酬なし、または極小に留める |
| 居場所 | アイドルと拠点の関係が近づく | 主目的として扱う |
| 物語 | 深層設定の入口を作る | 表層から不穏さの気配までに留める |

### `bond` との関係

- `bond` は、注目アイドルでライブした積み重ねとして扱う。
- 交流イベントの解放条件は、まず `idol.bond` Requirement を使う。
- 初期目安は `bond` 5、20、50 のような低・中・高の節目に分ける。
- 交流イベントは、`bond` の量そのものを消費しない。
- `bond` が増えた事実を、アイドルが場所や観客に慣れていく変化として見せる。

### `eventIdsRead` との関係

- `eventIdsRead` は、将来の交流イベント既読管理に使う。
- 既存の記録既読とは別管理にする。
- イベントIDは `idolId` と意味が分かる短い名前を組み合わせる。
  例: `otowaAkari.firstSeat`, `asagiriYui.lostCorner`, `mizukiShino.shelfGreeting`
- 既読済みイベントは再読できてよいが、初回読了の演出や報酬は1回だけにする。
- 最小実装では、既読状態の保存と表示済み判定を優先し、分岐や選択肢は入れない。

### イベント本文の方針

- プレイヤーは、アイドルを所有する人物ではなく、場所を一緒に戻す協力者として描く。
- 表層イベントでは、祈念工学、アンカー、祈念負荷、聖歌を使わない。
- `uncanny` 相当のイベントでも、核心設定を断定しない。
- 深層担当アイドルを追加するまでは、主人公の正体、霞の意志、旧計画の全貌を確定しない。
- 初期プロローグ組だけで物語を閉じず、後続アイドルや別区画へ余白を残す。

### 最小実装タスクの候補

1. `IdolEventDefinition` の型とコンテンツ定義ファイルを追加する。（実装済み）
2. `idol.bond` Requirement で解放される短い交流イベントを、音羽 灯里に1件だけ追加する。（実装済み）
3. `eventIdsRead` を使って既読保存を行う。（実装済み）
4. アイドルタブに「交流イベント」表示領域を追加する。（実装済み）
5. 初回読了時に数値を変えず、メッセージだけ更新する。（実装済み）
6. 動作確認後、朝霧 結、深月 詩乃へ1件ずつ増やす。（未実装）

現在の実装では、`otowaAkari.firstSeat` を音羽 灯里 交流 5 で解放する。これは交流イベント基盤の最小確認用であり、初期実装済みアイドル全員のイベントを揃えた状態ではない。

### 受け入れ条件

- `bond` の既存仕様と矛盾しない。
- `eventIdsRead` が保存・復元される。
- 交流イベントを読んでも、灯るさや生産量が予期せず変化しない。
- 通常UIで「聖歌」をシステム名にしていない。
- アイドルごとの個性は見えるが、プロローグ組だけで世界観核心を解決しない。

## 個性の扱い

「個性」は作品上重要になりうるが、序盤から数値資源として扱うと世界観核心に近づきすぎる。
当面は、説明文、記録、交流メモの中で各アイドルらしさを出すに留める。

将来的に資源化する場合も、主人公や灯し手の自己同一性に直結する正式仕様としていきなり確定しない。
まずは「交流」「親しさ」「担当区画」など、表向きのゲーム体験に近い語から導入する。

## Codex禁止事項

- 現在実装済みのアイドルだけで物語を完結させる設定を書かない。
- 新アイドルを追加するとき、世界観核心を一人に背負わせすぎない。
- 全新アイドルを全体生産倍率だけにしない。
- 深層担当アイドルを序盤に出しすぎない。
- fiction で承認済みの霞山 澪、七城 皐月、白霧 燐を、古い仮ロスター枠の未定候補へ戻さない。
