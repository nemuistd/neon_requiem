# アイドル拡張設計

## 位置づけ

この文書は、アイドルを将来的に10人以上へ増やすことを前提に、現在と今後の設計方針を整理する。

## 大前提

現在実装済みの3人は、物語全体を完結させるための全メンバーではない。
彼女たちはプロローグ組、または最初に世界の仕組みに触れる案内役として扱う。

## 現在の3人の位置づけ

### 音羽 灯里

- 初期アイドル。
- 路地裏ステージ、最初のライブ、最初の灯るさに関わる。
- プレイヤーに基本ループを教える。
- 世界観核心の解決者ではなく、最初の焦点。

### 朝霧 結

- ネオン掲示板、薄明通り、道案内、霞の濃い通路への導線。
- 序盤から中盤へ進む橋渡し。
- 地下礼拝堂より前に、霞や道の不安定さを自然に見せる役割を担える。

### 深月 詩乃

- 地下礼拝堂、失われた響き、保管された記録への導線。
- 深層要素に接続しやすいが、序盤で聖歌や祈念工学を解決する役ではない。
- 肩書きや説明は、通常UIで「聖歌係」に戻さない。

## 今後のアイドル分類

### 1. プロローグ組

現在の3人。
世界観と基本ループの入口を作る。

プロローグ組の役割は、ゲームの最初の理解を支えることである。
この3人だけで、霞、旧計画、主人公の正体、聖歌の全貌を解決しない。

### 2. 区画担当組

追加アイドルの主力。
各アイドルが新しい施設群・区画・プレイ要素に対応する。

候補例:

- 配信ブース系アイドル
- 記憶図書館系アイドル
- 地下駅跡 / 移動導線系アイドル
- 市場 / 広場系アイドル
- 修復工房系アイドル
- 記録保管区系アイドル
- ファンコミュニティ系アイドル

### 3. 機能担当組

プレイ要素の幅を広げるためのアイドル。
区画担当組と重なってよいが、効果設計では役割を明確にする。

候補例:

- ライブ獲得を伸ばす担当
- 特定施設群を伸ばす担当
- オフライン報酬を伸ばす担当
- アイテムや歌の取得を補助する担当
- 記録解放や既読導線を補助する担当
- 将来のファン資源や交流値を伸ばす担当

### 4. 深層担当組

中盤後半から後半に登場。
祈念工学、アンカー、記憶封印、旧計画などへ接続する。

深層担当組は、表向きの復興ゲームを壊さない範囲で断片を扱う。
一人のアイドルだけに世界観核心を背負わせない。

### 5. 転生後 / 再固定後組

転生 / 再固定 / 再観測後に意味が見えるアイドル。
1周目では名前や記録だけが見え、2周目以降に役割が変わって見える形も許容する。

転生後担当組は、最初から正式な仕様や正体を確定しない。
「1周目ではまだ意味が分からないが、後で読み替えられる」余白を残す。

## 仮ロスター枠

以下は名前やビジュアルを確定しない、役割だけの仮枠である。
今後のタスクで新アイドルを追加する場合は、この枠から必要な役割を選ぶ。

| 仮枠 | 主な接続先 | 表向きの役割 | 効果候補 |
| --- | --- | --- | --- |
| 配信ブース系 | 仮設配信ブース、告知、遠隔視聴 | 路地裏の外へ公演を届ける | ライブ獲得、ファン獲得、歌解放補助 |
| 記憶図書館系 | 記憶図書館、記録保管区 | 名前や記録を整理する | 記録解放補助、既読報酬、深層語の段階開示 |
| 地下駅跡 / 移動導線系 | 薄明通り、地下駅跡、道案内 | 消えかけた道をつなぐ | 施設解放補助、オフライン報酬、特定施設生産 |
| 市場 / 広場系 | 市場、広場、ファン待機所 | 人の流れと居場所を増やす | ファン獲得、アイテム価格軽減、施設生産 |
| 修復工房系 | 修復工房、設備保守 | 復興設備を支える | 施設強化補助、特定施設倍率、アイテム効果補助 |
| 記録保管区系 | 録音保管庫、名前の記録壁 | 音や名前を残す | 記録解放、歌効果補助、交流値獲得 |
| 深層担当 | 地下礼拝堂、深層施設 | 深い記録への入口 | 技術的断片の開示、転生後ヒント |
| 転生後担当 | 再固定後施設、再観測後記録 | 以前の周回を読み替える | 転生後ボーナス、持ち越し補助 |

この表は候補であり、全枠を必ず実装する必要はない。
また、1つのアイドルが複数の枠を兼ねてもよい。

## アイドル能力設計

10人以上になるため、全員を単純な「全灯るさ生産倍率」にしない。
全体倍率は分かりやすいが、人数が増えるほど効果が単調になり、注目アイドル選択の意味も薄くなる。

推奨する効果カテゴリ:

- 全灯るさ生産倍率
- ライブ獲得量
- オフライン報酬倍率
- 特定施設生産倍率
- アイテム価格軽減
- 歌 / アイテム解放補助
- 記録解放補助
- ファン獲得量
- 交流値獲得量
- 転生後ボーナス

効果設計の目安:

- プロローグ組は、基本ループの理解を助ける単純な効果でよい。
- 区画担当組は、対応する施設や区画に効果を寄せる。
- 機能担当組は、ライブ、施設、オフライン、記録、アイテムなどの軸を分担する。
- 深層担当組は、強力な数値効果よりも開示や条件補助を優先する。
- 転生後担当組は、1周目から必須にしない。

## `passiveEffects` と `focusEffects`

将来的には、アイドル定義を以下のように広げることを推奨する。

```ts
type IdolDefinition = {
  ...
  passiveEffects: Effect[];
  focusEffects?: Effect[];
}
```

- `passiveEffects`: 解放済みなら常時発動。
- `focusEffects`: 注目アイドルのときだけ発動。

これにより、注目アイドル選択がゲーム上の判断になる。

導入方針:

1. まず既存の `passiveEffect` と同じ挙動を `passiveEffects` 配列へ移す。
2. 既存3人の効果量や解放条件は変えない。
3. `focusEffects` は、注目アイドル選択の意味を持たせたい段階で追加する。
4. `focusEffects` を導入しても、注目していないアイドルが完全に無意味にならないようにする。
5. contentValidation で Effect 内の参照IDを検証する。

最初の実装時は大きなDSLを作らず、既存の Effect 型を拡張して扱う。

## アイドル効果モデル詳細設計

### 現在モデルの限界

現在のアイドル定義は、単一の `passiveEffect` を持つ。
既存3人は全員、解放済みなら常時発動する全灯るさ生産倍率である。

このモデルは最初の3人では分かりやすいが、10人以上に増えると以下の問題が出る。

- 全員が全体生産倍率になると、効果差が単調になる。
- 注目アイドルを切り替える意味が弱い。
- 特定施設、ライブ、オフライン報酬、記録解放などの役割分担がしにくい。
- 追加アイドルの個性を数値効果に反映しにくい。
- 後から `bond` や転生後効果を接続するとき、単数効果のままでは拡張しづらい。

### `passiveEffects: Effect[]`

`passiveEffects` は、アイドルが解放済みなら常時発動する効果配列とする。
既存の `passiveEffect` は、まず同じ挙動の1要素配列へ移行する。

初回移行のイメージ:

```ts
type IdolDefinition = {
  ...
  passiveDescription: string;
  passiveEffects: Effect[];
  focusEffects?: Effect[];
}
```

既存3人の移行方針:

- 音羽 灯里: 現在の全灯るさ生産 `x1.20` を維持する。
- 朝霧 結: 現在の全灯るさ生産 `x1.15` を維持する。
- 深月 詩乃: 現在の全灯るさ生産 `x1.10` を維持する。
- 解放条件、表示文、プレイ感は変えない。

### `focusEffects?: Effect[]`

`focusEffects` は、注目アイドルに選ばれている間だけ発動する効果候補である。
最初から効果を入れる必要はない。
型だけ先に入れるか、`passiveEffects` 移行後の別タスクに分ける。

導入する場合の用途:

- 注目中だけライブ獲得量を増やす。
- 注目中だけ特定施設の生産を伸ばす。
- 注目中だけ `bond` 獲得量を増やす。
- 注目中だけ記録解放やアイテム解放を補助する。

注意点:

- 注目していないアイドルの価値を消さない。
- 頻繁な切り替えが必須になるほど強くしない。
- 既存の全体パッシブと乗算順が分かりにくくならないようにする。
- UIでは、常時効果と注目効果を別ラベルで表示できる余地を残す。

### 効果カテゴリ

10人以上に増やす場合、以下のカテゴリから役割を分散する。

| カテゴリ | 常時効果向き | 注目効果向き | 備考 |
| --- | --- | --- | --- |
| 全灯るさ生産倍率 | ○ | △ | 使いすぎると単調になる |
| ライブ獲得量 | ○ | ○ | 注目アイドルの手触りを出しやすい |
| 特定施設生産倍率 | ○ | ○ | 区画担当組と相性がよい |
| オフライン報酬倍率 | ○ | △ | 導線施設、道案内系と相性がよい |
| アイテム価格軽減 | ○ | △ | バランス影響が大きいので控えめにする |
| 歌 / アイテム解放補助 | ○ | ○ | Requirement と contentValidation の検証が必要 |
| 記録解放補助 | ○ | ○ | 深層開示を急がないよう注意する |
| ファン獲得量 | ○ | ○ | ファン資源導入後の候補 |
| bond 獲得量 | ○ | ○ | 交流値導入後の候補 |
| 転生後ボーナス | ○ | △ | 1周目必須にしない |

### 実装PR分割案

1. 型移行だけを行う。
   `IdolDefinition.passiveEffect` を `passiveEffects: Effect[]` へ変更し、既存3人の効果を同じ数値で配列化する。
2. 集計関数を置き換える。
   既存のアイドル全体生産倍率計算を、解放済みアイドルの `passiveEffects` を集約する形へ変更する。
3. contentValidation を更新する。
   アイドル定義内の Effect 参照IDや数値妥当性を検証対象に含める。
4. UI表示は一旦 `passiveDescription` を維持する。
   効果配列から表示文を自動生成しない。
5. 必要なら `focusEffects?: Effect[]` を型だけ追加する。
   実効果の利用は、注目アイドルの意味を増やす別PRに回してよい。
6. focusEffects 実装時に、常時効果と注目効果の表示、集計、テストを追加する。

### テスト方針

移行PRでは、少なくとも以下を確認する。

- 既存3人の生産倍率が移行前後で変わらない。
- 未解放アイドルの `passiveEffects` は発動しない。
- 解放済みアイドルの複数効果がすべて集計される。
- `focusEffects` を導入した場合、注目アイドル以外では発動しない。
- `contentValidation` が不正な Effect 数値や参照IDを検出する。

## 交流値

実装名は `bond` を第一候補にする。
UI名は「交流」または「親しさ」とする。

導入順:

1. `GameState.idols[id].bond` のような個別状態を追加する。
2. 注目アイドルでライブしたときだけ、少量の bond を増やす。
3. UIに「交流」または「親しさ」として表示する。
4. `idol.bond` Requirement を追加し、記録や小さなイベント解放に使う。
5. その後、bond 獲得量や focusEffects との接続を検討する。

注意:

- 「信頼」は、将来の人間関係表現として使う余地はあるが、灯し手本人の自己同一性維持資源にはしない。
- 「個性」は重要概念だが、序盤から本格資源化しない。
- 最初はbondを、注目アイドルとの交流値として扱う。

## 個性の扱い

「個性」は作品上重要になりうるが、序盤から数値資源として扱うと世界観核心に近づきすぎる。
当面は、説明文、記録、交流メモの中で各アイドルらしさを出すに留める。

将来的に資源化する場合も、主人公や灯し手の自己同一性に直結する正式仕様としていきなり確定しない。
まずは「交流」「親しさ」「担当区画」など、表向きのゲーム体験に近い語から導入する。

## Codex禁止事項

- 現在の3人だけで物語を完結させる設定を書かない。
- 新アイドルを追加するとき、世界観核心を一人に背負わせすぎない。
- 全新アイドルを全体生産倍率だけにしない。
- 深層担当アイドルを序盤に出しすぎない。
