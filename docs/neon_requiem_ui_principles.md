# NEON REQUIEM UI設計方針

## 位置づけ

この文書は、NEON REQUIEM のUIを今後拡張していく際に、画面構成・情報優先度・キャラクター表示の方針がぶれないようにするための設計原則である。

世界観・用語体系は [worldbuilding.md](worldbuilding.md) を正史とする。現行仕様は [current_spec.md](current_spec.md) を参照する。

## 1. 基本方針

NEON REQUIEM は、単なる数値増加ゲームではなく、地下都市・ネオン・歌・アイドルを軸にしたインクリメンタルゲームである。

UIでは以下を両立する。

- インクリメンタルゲームとして必要な数値・操作・強化の見やすさ
- アイドルゲームとしてのキャラクター存在感
- 地下都市、ネオン、歌、少しダークな雰囲気
- 将来の施設・アップグレード追加に耐える拡張性
- メッセージによる世界観表現と進行フィードバック

序盤UIでは、祈念工学、アンカー、祈念負荷、聖歌などの深層語を前面に出しすぎない。表向きには、アイドル拠点運営と復興ゲームとして始める。

## 2. 基本レイアウト

現在の基本レイアウトは以下。

```text
[1] 上部ヘッダー / 操作パネル
[2] メッセージバー
[3] メイン領域
    左: 注目アイドル枠
    右: タブ切替式コンテンツ領域
[4] 設定パネル
```

重要なのは、施設・歌・アイテム・アイドル・記録を同じ縦長リストに混ぜないこと。

現在のタブ:

- 復興区画
- 歌
- アイテム
- アイドル
- 記録
- 廻（再固定中枢解放後）

## 3. スクロールに対する考え方

スクロール自体は許容する。

施設、歌、アイテム、未解放要素、記録などは増えるため、すべてを1画面に収めようとするとUIが破綻する。

ただし、スクロールによって以下が見えなくなる状態は避ける。

- 現在の主要リソース
- 秒間生産量
- 主要操作ボタン
- 重要な最新メッセージ
- 少なくとも1人の注目アイドル

## 4. 上部ヘッダー / 操作パネル

上部ヘッダーは、ゲームを操作するための最重要領域である。

表示するもの:

- コンパクトなタイトル: NEON REQUIEM
- 短いサブタイトル
- 灯るさ
- 毎秒生産量
- ライブするボタン
- 設定ボタン

演出文として「灯り」は使ってよいが、数値リソース表示は「灯るさ」に統一する。

設定ボタンは右上に1つだけ置く。左下などに重複して置かない。

## 5. メッセージバー

メッセージは、単なる通知ではなく世界観表現と進行フィードバックを担う。

メッセージバーは上部ヘッダー直下に横長で配置する。

表示内容の例:

- ライブ成功
- 施設強化
- 歌購入
- アイテム購入
- 記録既読
- オフライン報酬
- 新要素解放の短い反応

オートセーブ成功は頻繁に発生するため、メインメッセージを上書きしない。

## 6. 注目アイドル枠

常に表示するのは「現在の注目アイドル」1人を基本とする。

表示内容:

- 名前
- ふりがな
- 肩書き
- 短いフレーバーテキスト
- パッシブ効果
- 詳細ボタン
- アイドル切替候補

初期の注目アイドルは **音羽 灯里（おとは あかり）**。

現在は注目アイドルの選択状態を保存する。未解放アイドルは、関連施設がまだ見えていない段階では候補として表示しない。関連施設が見えた後だけ、汎用名のロックカードとして存在を示し、未解放状態では選択できない。

## 7. 右側コンテンツ領域

右側コンテンツ領域は、タブ切替式で扱う。

### 復興区画

施設・区画を表示する。

現在の施設:

- 路地裏ステージ
- ネオン掲示板
- 薄明通り案内所
- 仮設配信ブース
- 記憶図書館
- 録音保管庫
- 古い放送室
- 地下広場
- 名前の記録壁
- 地下礼拝堂
- 地下通路修復区画
- 再固定中枢
- 深層観測所
- 工学記録保管区
- 祈念工学実験跡地
- 再観測拠点
- 名前のない劇場

地下礼拝堂は序盤施設ではなく、名前の記録壁の後に出る中盤以降の深層施設として扱う。
地下通路修復区画は、地下礼拝堂 Lv8 と礼拝堂のハーモニー取得後に表示される深層接続施設として扱う。再固定中枢は、地下通路修復区画 Lv5 と修復の仮歌取得後に表示される廻本体の入口として扱う。
深層観測所、工学記録保管区、祈念工学実験跡地は初回廻後、再観測拠点と名前のない劇場は廻2以降に表示される。

### 歌

1回購入型の歌アップグレードを表示する。

現在の歌:

- 路地裏のイントロ
- 配信前夜のアカペラ
- 記録の歌
- 広場のアンセム
- 礼拝堂のハーモニー
- 薄明のコーラス
- 修復の仮歌
- 最後の名前

「聖歌」は通常タブ名や通常システム名にしない。

### アイテム

1回購入型の小〜中規模アップグレードを表示する。アイテムタブは実装済みとして扱う。

現在のアイテム:

- 古いネオン管
- 手書きの告知ポスター
- 半券の束
- 携帯スポットライト
- 録音済みの短い挨拶
- 交代用連絡ボード
- 古い電波塔の残骸
- 手書きのリスナー名簿
- 色あせた書名ラベル
- 放送室の機材マニュアル
- 修復用の工具一式

### アイドル

解放済みアイドル、未解放アイドル、各アイドルの概要を表示する。

全アイドルを常時メイン画面に大きく並べるのではなく、このタブで一覧管理する。

### 記録

世界観ログ、進行記録、主人公の断片的な記憶を扱う。

記録には数値効果を持たせず、既読状態と追記既読状態だけを保存する。revealLevel により、surface / uncanny / technical / deep の開示段階を管理する。廻後追記は本文差し替えではなく、末尾に別色の追記として表示する。

### 廻

再固定中枢解放後に表示する。初回前は廻実行パネルと記憶断片の獲得見込みを中心にし、初回後は記憶断片、購入済みバフ、候補バフを確認できるようにする。

廻は深層システムだが、UI上は「リセットされるもの」「引き継ぐもの」を明確にし、プレイヤーが何を失い何を残すかを判断できることを優先する。

## 8. 施設・アップグレードカード

施設カードには最低限以下を表示する。

- 施設名
- Lv
- 強化コスト
- 生産量
- 強化ボタン
- 未解放時の解放条件

歌・アイテムカードには最低限以下を表示する。

- 名前
- 説明
- 効果説明
- コスト
- 解放条件
- 購入ボタン
- 購入済み状態

未解放要素は、ネタバレを避けるため段階的に存在を示す。復興区画は次に到達できる1件だけを汎用名「未確認区画」として表示する。歌・アイテム・アイドルは、関連施設が見えるまではカード自体を表示せず、関連施設が見えてからは汎用名のロックカードとして表示する。記録は未解放時点ではタイトルも条件も表示しない。

## 9. 左下メニューに対する方針

デスクトップ画面では、左下に独立したメニューを置く方針は採用しない。

画面切替やカテゴリ移動は、右側コンテンツ領域のタブで扱う。設定は右上に1つだけ置く。

## 10. 美少女要素の表示方針

NEON REQUIEMでは、アイドル要素を数値ゲームの装飾にしない。

方針:

- 常に少なくとも1人のアイドルを表示する。
- 注目アイドルは大きめに表示する。
- 全員を常時小さく並べることは避ける。
- 施設やアップグレードとアイドルの関係を持たせる。
- アイドルは単なる生産装置にしない。

## 11. 操作性と雰囲気のバランス

望ましい印象:

- 暗い地下都市
- ネオンの光
- 歌、ステージ、復興区画
- 少しダークだが美しい
- アイドルの存在感

ただし、雰囲気演出が以下を妨げてはいけない。

- 文字の読みやすさ
- ボタンの押しやすさ
- コストや生産量の確認
- 強化可能かどうかの判断
- 最新メッセージの視認性

## 12. スマホ幅の最低方針

スマホ幅では、左右分割ではなく縦積みを基本とする。

優先順:

1. 上部ヘッダー / 操作パネル
2. メッセージバー
3. 注目アイドル枠
4. タブ切替式コンテンツ領域

本格的なモバイル最適化はまだ未実装だが、大きく壊さないことを優先する。

## 13. 実装済みのUI要素

- コンパクトな上部ヘッダー
- 灯るさ、毎秒、ライブするボタン、設定ボタン
- ヘッダー直下のメッセージバー
- 注目アイドル枠
- 注目アイドル詳細パネル
- アイドル切替候補
- 関連施設が見えた未解放アイドルの汎用ロック表示
- 復興区画 / 歌 / アイテム / アイドル / 記録のタブ
- 再固定中枢解放後の廻タブ
- 廻実行パネル、記憶断片表示、廻後バフ一覧
- アイドルタブと詳細パネル内の交流イベント表示
- 設定パネル
- セーブ削除ボタン

## 14. まだ行わない改善

- 立ち絵制作や画像追加の大規模作業
- スマホ専用の細かい最適化
- 詳細な過去ログ機能
- 複雑なアニメーション付きヘッダー
- 深層用語を序盤UIに大きく出す演出
- 主人公の正体、霞の意志、旧計画の全貌を確定する演出

## 15. Codexへの実装時注意

CodexがUIを変更する場合は、以下を守ること。

- ゲームロジック、保存処理、数値計算を不用意に変更しない。
- 新機能を不用意に追加しない。
- 施設、歌、アイテム、アイドル、記録の役割を混ぜない。
- 常に少なくとも1人の注目アイドルが見える構成を維持する。
- リソース、毎秒、ライブするボタン、メッセージバーを見失いにくい位置に置く。
- 設定ボタンを複数箇所に置かない。
- 左下メニューを安易に追加しない。
- 「聖歌」を通常システム名にしない。
- 序盤UIに祈念工学、アンカー、祈念負荷、聖歌などを出しすぎない。

大きなレイアウト変更を行う場合は、実装前に変更方針を説明すること。

## 16. historical: 旧UI案の扱い

過去のUI案には「歌 / アイテム」統合タブ、アイテム未実装、地下礼拝堂を早期に扱う例、リソース名「灯り」などが含まれていた。

現在は、歌タブとアイテムタブを分け、アイテムタブは実装済みとして扱う。数値リソース表示は「灯るさ」に統一する。
