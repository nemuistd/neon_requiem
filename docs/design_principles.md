# NEON REQUIEM 設計原則

## 目的

この文書は、NEON REQUIEM の拡張で、ゲーム要素の役割が混乱しないようにするための設計原則をまとめる。

これは最終仕様書ではない。実装とプレイ感を見ながら更新する。

世界観・用語体系の判断は `docs/worldbuilding.md` を優先する。特に「歌」「灯し手」「アンカー」「霞」「祈念工学」「灯るさ」「個性」の扱いは、worldbuilding の方針に従う。

## 基本分類

NEON REQUIEM では、ゲーム要素を以下のように分ける。

- アイドル
- 施設 / 区画
- 歌
- アイテム
- 交流
- リソース
- メッセージ / 記録
- 将来の転生 / 深層巡礼要素

## 施設 / 区画

施設は、地下都市の復興対象であり、原則としてリソースを直接生産する。

現在の施設:

- 路地裏ステージ: 灯りを直接生産する。
- 地下礼拝堂: 灯りを直接生産する。

原則:

- 施設はレベルを持つ。
- 施設は購入・強化できる。
- 施設は灯りを生産する基本施設として扱う。
- 全体倍率だけを主機能にする施設は避ける。
- 区画ごとの複雑な役割分担は行わない。

## 歌 / アイテム

倍率効果や特殊効果は、原則として施設ではなく歌またはアイテム側に置く。

現在の歌:

- 路地裏のイントロ: ライブ1回の灯り +1
- 礼拝堂のハーモニー: 施設の灯り生産 x1.10
- 薄明のコーラス: 施設の灯り生産 x1.25

原則:

- 歌は大型アップグレードとして扱い、条件達成で解放され、恒久効果を持つ。
- 歌は1回購入型を基本とし、購入済みなら効果が常時発動する。
- アイテムは通常アップグレードとして扱い、小〜中規模の恒久効果を持つ。
- 深層用語としての「聖歌」は通常システム名にしない。
- Phase 2 時点では、歌システムは実装済み。アイテムシステムはまだ未実装。

## アイドル

アイドルは、世界観、全体パッシブ効果、交流イベントを担う中心要素である。

現在のアイドル:

- 音羽 灯里: 初期ライブ、灯り、路地裏の復興に関係する。
- 深月 詩乃: 地下礼拝堂、歌、記録に関係する。
- 朝霧 結: 霞の濃い通路、案内、後続区画への導線に関係する。

原則:

- アイドルの担当区画システムは採用しない。
- アイドルは解放済みなら全体パッシブ効果を持つ。
- アイドルには将来的に交流イベントを追加する。
- アイドル追加は、単なる数値追加ではなく、世界観や解放要素の追加とセットにする。
- アイドルは単なる生産装置にしない。
- 現時点では複雑な編成システムにはしない。
- 注目アイドルは1人を大きく表示し、候補は小さな切替リストで扱う。

## リソース

現在のリソースは灯りのみ。

将来的な候補:

- 応援
- 残響
- 灯るさ
- 記憶片
- 個性

原則:

- リソース追加は慎重に行う。
- 新リソースは、新しい判断や進行を生む場合だけ追加する。
- 単なる名前違いの通貨を増やさない。

## UI分類

現在のUIは以下の分類を基本にする。

- 上部ヘッダー / 操作パネル
- ヘッダー直下のメッセージバー
- 左側の注目アイドル枠
- 右側のタブまたはカテゴリ切替式コンテンツ領域

原則:

- メッセージバーはヘッダー直下に大きめに置く。
- 右側コンテンツは将来的にタブまたはカテゴリ切替にする。
- デスクトップでは左下固定メニューは採用しない。
- 設定は1か所に統一する。
- 注目アイドル枠は表示だけでなく、パッシブ効果や交流導線を示す意味のある情報枠にする。
- 施設・歌・アイドル・記録を混ぜずに分類する。

## 記録

記録は、ゲーム進行に応じて解放される読み物要素である。

現在の記録:

- 路地裏ステージ復旧メモ
- 観測記録・灯り反応
- 地下礼拝堂復旧報告
- 歌と聖歌の区別について
- 施設ログ・霞とアンカー

原則:

- 記録は灯り、生産量、強化コストに影響しない。
- 記録は世界観、主人公の忘れていたこと、新区画で得た知識を開示する。
- 記録には解放条件を持たせる。
- 記録の既読状態は保存する。
- 深層用語は記録内で慎重に扱い、通常システム名と混ぜない。

## 保存とテスト

保存は `localStorage` を使う。

現在の保存キー:

```text
neon-requiem-save-v1
```

現在のセーブデータバージョン:

```text
saveVersion: 5
```

原則:

- 通常プレイでは短い間隔で自動保存してよい。
- オートセーブ成功通知は、メインメッセージを上書きしない。
- 手動テストで `localStorage` を直接編集すると、アプリ内メモリ状態に上書きされる可能性がある。
- テスト用セーブ投入には `window.__NEON_DEBUG__` を使う。
- コンテンツ定義の整合性確認には `npm run check:content` を使う。

## Codexへの注意

Codexは、今後の実装でこの分類を優先すること。

施設、アップグレード、アイドル、リソース、記録の役割を混ぜない。

新機能を追加する場合は、それがどのカテゴリに属するのかを先に説明すること。
