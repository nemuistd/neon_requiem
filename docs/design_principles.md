# NEON REQUIEM 設計原則

## 位置づけ

この文書は、NEON REQUIEM の拡張でゲーム要素の役割が混乱しないようにするための設計原則をまとめる。

世界観・用語体系の判断は [worldbuilding.md](worldbuilding.md) を正史として優先する。実装済み仕様の詳細は [current_spec.md](current_spec.md)、手動確認は [manual_qa_checklist.md](manual_qa_checklist.md) を参照する。

## 基本分類

現在のゲーム要素は以下に分ける。

- アイドル
- 施設 / 復興区画
- 歌
- アイテム
- リソース
- メッセージ / 記録
- 交流イベント
- 廻 / 深層巡礼要素

施設、歌、アイテム、アイドル、記録の役割を混ぜすぎない。数値効果を持つものは、施設・歌・アイテム・アイドルパッシブとして明確に扱う。

## 施設 / 復興区画

施設は地下都市の復興対象であり、原則として「灯るさ」を直接生産する。

現在の施設:

- 路地裏ステージ: 初期施設。Lvごとに `0.1 灯るさ / 秒`。
- ネオン掲示板: 序盤2施設目。路地裏ステージ Lv10 で解放。Lvごとに `0.35 灯るさ / 秒`。
- 薄明通り案内所、仮設配信ブース、記憶図書館、録音保管庫、古い放送室、地下広場、名前の記録壁: 地下礼拝堂へ急ぎすぎないための導線・記憶施設群。
- 地下礼拝堂: 中盤以降の深層施設。名前の記録壁 Lv3 で解放。Lvごとに `16 灯るさ / 秒`。
- 地下通路修復区画: 地下礼拝堂 Lv8 と礼拝堂のハーモニー取得後の深層接続施設。Lvごとに `20 灯るさ / 秒`。
- 再固定中枢: 地下通路修復区画 Lv5 と修復の仮歌取得後の廻起点施設。Lvごとに `25 灯るさ / 秒`。
- 深層観測所、工学記録保管区、祈念工学実験跡地: 初回廻後に見える Ch.7〜Ch.8 の深層施設群。
- 再観測拠点、名前のない劇場: 廻2以降に見える Ch.9 起点施設群。

原則:

- 施設はレベルを持つ。
- 施設は解放条件を持てる。
- 施設は強化コストを消費してレベルアップする。
- 地下礼拝堂は削除しないが、序盤2施設目として扱わない。
- 施設説明では、序盤から祈念工学、アンカー、祈念負荷、聖歌などの深層語を前面に出しすぎない。

## 歌 / アイテム

歌とアイテムは、1回購入型の恒久アップグレードとして扱う。

現在の歌:

- 路地裏のイントロ: 路地裏ステージ Lv5 で解放。ライブ1回の灯るさ `+1`。
- 配信前夜のアカペラ: 仮設配信ブース Lv2 で解放。ライブ1回の灯るさ `+8`。
- 記録の歌: 紙野 巡 合流済み、記憶図書館 Lv3 で解放。施設の灯るさ生産 `x1.15`。
- 広場のアンセム: 陽向 小春 合流済み、地下広場 Lv3 で解放。ライブ1回の灯るさに施設生産の一部を加算。
- 礼拝堂のハーモニー: 地下礼拝堂 Lv2 で解放。施設の灯るさ生産 `x1.10`。
- 薄明のコーラス: 地下礼拝堂 Lv8 で解放。施設の灯るさ生産 `x1.25`。
- 修復の仮歌: 継ノ端 桜子 合流済み、地下通路修復区画 Lv3 で解放。オフライン灯るさ報酬 `x1.10`。
- 最後の名前: 廻2以降、白霧 燐 合流済み、名前のない劇場 Lv2 で解放。施設生産と記憶断片の獲得見込みを補強する Ch.9 収束補強の歌。

現在のアイテム:

- 古いネオン管: 路地裏ステージ Lv3 で解放。施設の灯るさ生産 `x1.05`。
- 手書きの告知ポスター: 路地裏ステージ Lv4 で解放。ライブ1回の灯るさ `+1`。
- 半券の束: 路地裏ステージ Lv5 で解放。ライブ1回の灯るさ `+1`。
- 携帯スポットライト: 路地裏ステージ Lv6 で解放。施設の灯るさ生産 `x1.04`。
- 録音済みの短い挨拶: ネオン掲示板 Lv3 で解放。施設の灯るさ生産 `x1.03`。
- 交代用連絡ボード: ネオン掲示板 Lv2 で解放。オフライン灯るさ報酬 `x1.10`。
- 古い電波塔の残骸、放送室の機材マニュアル: 配信・記憶施設帯の中盤アイテム。
- 手書きのリスナー名簿: 響木 遠子 合流済み、仮設配信ブース Lv4 で解放。施設の灯るさ生産 `x1.06`。
- 色あせた書名ラベル: 紙野 巡 合流済み、記憶図書館 Lv3 で解放。交流増加量 `x1.10`。
- 修復用の工具一式: 継ノ端 桜子 合流済み、地下通路修復区画 Lv2 で解放。施設の灯るさ生産 `x1.08`。

オフライン報酬倍率は、同じ時間のオンライン自然生産に対する基礎効率50%に掛ける補助効果として扱う。`交代用連絡ボード` は実効55%で、今後の追加効果もオンライン自然生産を超える前提にしない。

原則:

- 「聖歌」は通常システム名にしない。
- 通常UIでは「歌」「楽曲」「公演」「演目」「アップグレード」などを使う。
- アイテムタブは実装済みとして扱う。
- アイテムは小〜中規模の恒久効果を担う。
- Effect共通型に載る範囲で、効果の追加は小さく保つ。

## アイドル

アイドルは、世界観、全体パッシブ効果、交流イベントを担う中心要素である。

現在のアイドル:

- 音羽 灯里: 初期アイドル。全灯るさ生産 `x1.20`。
- 朝霧 結: ネオン掲示板 Lv5 で解放。全灯るさ生産 `x1.15`。
- 響木 遠子: 仮設配信ブース Lv3 で解放。ライブ1回の灯るさに施設生産の一部を加算。
- 深月 詩乃: 地下礼拝堂 Lv3 で解放。全灯るさ生産 `x1.10`。
- 紙野 巡: 記憶図書館 Lv2 で解放。全アイドルの交流増加量を補強。
- 陽向 小春: 地下広場 Lv1 で解放。施設生産とアイテム購入コストを補強。
- 継ノ端 桜子: 地下通路修復区画 Lv1 で解放。オフライン灯るさ報酬 `x1.15`。
- 霞山 澪: 初回廻後、深層観測所 Lv1 で解放。deepタグ施設の生産を補強。
- 七城 皐月: 初回廻後、工学記録保管区 Lv3 で解放。歌・アイテム取得コストを補助。
- 白霧 燐: 廻2以降、名前のない劇場 Lv1 で解放。廻後生産倍率と記憶断片の獲得見込みを補強。

原則:

- アイドルは単なる生産装置にしない。
- アイドルは解放済みなら全体パッシブ効果を持つ。
- 注目アイドルは1人を大きく表示し、候補は小さな切替リストで扱う。
- 「灯し手」は表向き・人間的な呼称、「アンカー」は旧工学側の管理的な呼称として扱う。
- 序盤UIでは「アンカー」をアイドルの通常名称にしない。

## リソース

現在の数値リソースは `resources.tomorusa` で、表示名は「灯るさ」。

原則:

- 数値リソースとしては「灯るさ」を使う。
- 演出文としての「灯り」は残してよい。
- 複数リソース追加は慎重に行う。
- 新リソースは、新しい判断や進行を生む場合だけ追加する。
- 単なる名前違いの通貨を増やさない。

## UI分類

現在のUIは以下の分類を基本にする。

- 上部ヘッダー / 操作パネル
- ヘッダー直下のメッセージバー
- 左側の注目アイドル枠
- 右側のタブ切替式コンテンツ領域
- 設定パネル

現在のタブ:

- 復興区画
- 歌
- アイテム
- アイドル
- 記録
- 廻

原則:

- 施設、歌、アイテム、アイドル、記録を混ぜずに分類する。
- メッセージバーはヘッダー直下に置く。
- 設定は1か所に統一する。
- 左下固定メニューは採用しない。
- 右側コンテンツはタブ切替で拡張する。

## 記録

記録は、ゲーム進行に応じて解放される読み物要素である。

現在の記録は `surface`, `uncanny`, `technical`, `deep` の revealLevel を持つ。

原則:

- 記録は灯るさ、生産量、強化コストに影響しない。
- 記録は世界観、主人公の忘れていたこと、新区画で得た知識を開示する。
- 記録には解放条件を持たせる。
- 記録の既読状態は保存する。
- surface の記録では、祈念工学、アンカー、聖歌、祈念負荷などの深層語を出さない。
- deep / technical の記録でも、通常システム名と深層用語を混ぜない。

## 保存とテスト

保存は `localStorage` を使う。

現在の保存キー:

```text
neon-requiem-save-v1
```

現在のセーブデータバージョン:

```text
saveVersion: 11
```

現在の保存構造は `resources.tomorusa`, `resources.memoryFragment`, `totalTomorusaEarned`, `facilities`, `idols`, `items`, `songs`, `records`, `meguri`, `activeIdolId`, `recordTabLastSeenContentVersion`, `lastSavedAt` を持つ。

原則:

- 通常プレイでは短い間隔で自動保存してよい。
- オートセーブ成功通知は、メインメッセージを上書きしない。
- 壊れたセーブでアプリがクラッシュしないようにする。
- テスト用セーブ投入には `window.__NEON_DEBUG__` を使う。
- コンテンツ定義の整合性確認には `npm run check:content` を使う。

## 現時点で意図的に未実装のもの

- 複数リソース
- ガチャ
- 課金
- 音声
- 複雑なストーリーシステム
- 詳細なセーブ復旧UI

## Codexへの注意

Codexは、今後の実装でこの分類を優先すること。

- 新機能を追加する場合は、それがどのカテゴリに属するのかを先に説明する。
- 世界観の重大設定は [worldbuilding.md](worldbuilding.md) にないものを勝手に確定しない。
- 「聖歌」を通常システム名にしない。
- 序盤UIに祈念工学、アンカー、祈念負荷、聖歌などを出しすぎない。
- ゲーム体験の中心は搾取や絶望ではなく、回復、居場所づくり、復興に置く。
