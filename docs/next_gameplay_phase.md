# NEON REQUIEM 次ゲーム内容拡張案

## 位置づけ

この文書は、v0.2基盤後に次に実装するゲーム内容拡張の設計メモである。
この段階では実装しない。世界観の正史は [worldbuilding.md](worldbuilding.md) を優先し、未記載の重大設定はここでは確定しない。

次に実装する候補としては、まず「アイドル交流MVP」を推奨する。

理由:

- 注目アイドル機能に、表示以外の意味を持たせられる。
- アイドルが単なる全体パッシブ効果だけの存在になりにくい。
- 「名前を呼ばれる」「顔を覚えられる」「個性を保つ」という世界観方針と相性がよい。
- 新しい数値資源を増やさず、現在の `resources.tomorusa` と既存ライブ操作の延長で実装できる。

## アイドル交流MVP案

### 基本ループ

1. プレイヤーが注目アイドルを選ぶ。
2. その状態で「ライブする」を押す。
3. 灯るさに加えて、注目アイドルの交流値が少し増える。
4. 一定値に達すると短い交流メモが解放される。
5. 交流メモを読むことで、アイドル本人や復興拠点の小さな変化が描写される。

最初は、交流値は単純な数値だけでよい。
交流値はアイドルを支える時間や、名前を呼び続ける積み重ねとして扱い、搾取的な表現にしない。

### 用語

実装名は `bond` を第一候補とする。
UI表示名は「交流」または「親しさ」程度に留める。

「信頼」は、人間関係の言葉として将来使う余地はあるが、灯し手本人の自己輪郭を保つ資源名としては使わない。
「個性」は重要概念だが、MVPでは本格的な資源にはしない。

## State案

現在の `GameState` には、アイドルごとの状態がない。
導入する場合は、セーブ構造に以下を追加する。

```ts
idols: Record<IdolId, {
  bond: number;
  eventIdsRead: string[];
}>
```

最小実装では、`eventIdsRead` をすぐ使わず、既存の `records` に交流メモを載せる案もありうる。
ただし、将来的に交流イベントを通常記録と分けるなら、`eventIdsRead` を最初から用意しておくと移行が軽い。

セーブ影響:

- `SAVE_VERSION` の更新が必要になる可能性が高い。
- 旧セーブ読み込み時は、全アイドルの `bond: 0`, `eventIdsRead: []` を補完する。
- 壊れたセーブでクラッシュしない方針は維持する。
- `window.__NEON_DEBUG__` と設定画面のセーブ削除は維持する。

## Unlock案

最初の交流メモは短く、表向きの復興ゲームとして自然な内容にする。
深層語は、序盤交流メモでは前面に出さない。

候補:

| 条件 | 解放内容 | 方針 |
| --- | --- | --- |
| 音羽 灯里 bond 5 | 交流メモ1 | 路地裏ステージで名前を呼ばれることへの反応 |
| 音羽 灯里 bond 20 | 交流メモ2 | 小さな常連ができ始めた描写 |
| 朝霧 結 bond 5 | 交流メモ1 | ネオン掲示板や道案内に関する短いメモ |
| 深月 詩乃 bond 5 | 交流メモ1 | 地下礼拝堂の保管物に関する短いメモ |

深月詩乃は地下礼拝堂に合わせた後方解放のまま扱う。
朝霧結はネオン掲示板の後に自然に関わる導線として扱う。

## UI案

### 注目アイドル枠

- 現在の注目アイドル枠に交流値を追加表示する。
- 表示例: `交流 3`
- 詳細ボタンはMVPでは無効のままでよい。
- 将来、詳細ボタンから交流メモ一覧や個別イベントへ進む導線にできる。

### アイドルタブ

- 各アイドルカードに交流値を表示する。
- 未解放アイドルは、交流値ではなく解放条件を優先して見せる。
- 解放済みアイドルは、注目アイドルへ切り替えた上でライブすると交流値が増えることが分かる程度でよい。

### 記録タブとの関係

選択肢は2つある。

1. 通常の記録にカテゴリ「交流メモ」を追加する。
2. 将来のアイドル詳細画面用に、通常記録とは別の `idolEvents` として保持する。

MVPでは1を推奨する。
理由は、既存の記録解放・既読保存・タブ表示を利用でき、UI追加が小さく済むため。

ただし、後でアイドル詳細画面を作る場合に移動できるよう、record id は `idolBond...` のように由来が分かる命名にする。

## Effect / Requirement案

交流メモ解放には、既存のRequirementに新型を足す案がある。

```ts
{ type: "idol.bond", idolId: "otowaAkari", amount: 5 }
```

MVPでは、Requirementを大きなDSLにしない。
既存の `facility.level`, `song.purchased`, `resource.amount` と同じ粒度の小さな追加に留める。

Effectは、この段階では追加しなくてよい。
交流メモは数値効果を持たず、読み物として扱う。

## 実装PR分割案

### PR 1: Stateと保存の下準備

- `GameState` に `idols` 状態を追加
- `createInitialState` に初期値を追加
- `storage.ts` で旧セーブ補完を追加
- `SAVE_VERSION` を上げるか判断する
- ロジックテストを追加

確認:

- 新規セーブで全アイドル `bond: 0`
- 旧セーブで `idols` が補完される
- 壊れたセーブでクラッシュしない

### PR 2: ライブによる交流値加算

- 注目アイドル状態でライブしたとき、対象アイドルの `bond` を増やす
- 最初はライブ1回につき `+1`
- 手動ライブのみ対象。自動生産では増やさない
- `gainManualTomorusa` 周辺のロジックテストを追加

確認:

- 注目アイドルを切り替えると、加算先が変わる
- 未解放アイドルには切り替えられない既存仕様を維持
- 既存の灯るさ獲得量は変わらない

### PR 3: UI表示

- 注目アイドル枠に交流値を表示
- アイドルタブに交流値を表示
- 詳細ボタンはまだ無効のまま維持
- CSSは最小変更に留める

確認:

- 表示が既存カードから大きく崩れない
- モバイル表示を大きく壊さない

### PR 4: 交流メモ解放

- `Requirement` に `idol.bond` を追加
- contentValidationで参照IDとamountを検証
- 記録カテゴリに「交流メモ」を追加するか、既存カテゴリ内で扱うかを決める
- 音羽灯里のbond 5 / 20、朝霧結 bond 5、深月詩乃 bond 5 の短いメモを追加

確認:

- 交流値到達で記録が解放される
- 記録を読んでも灯るさや生産量は変わらない
- surface相当の交流メモに深層語が出すぎない

## 採用しない案

### ファン資源MVPを先に入れる

ファン資源は世界観と相性がよいが、複数リソース、UI表示、バランス、保存、効果設計が同時に増える。
次の一手としては、アイドル交流より影響範囲が大きい。

### 記録/章進行MVPを先に入れる

章進行は良い拡張だが、現時点ではゲーム操作より読み物管理の比重が増えやすい。
まず交流値を入れることで、記録解放の新しい軸を作ってから章進行へ進める方が自然。

### 薄明通り系の新施設を先に入れる

新施設追加はバランス調整とコンテンツ追加が大きくなる。
朝霧結の導線には向いているが、v0.2直後は既存アイドルの意味づけを先に強める方が安全。

## 次に実装すべき内容

最初の実装PRは「PR 1: Stateと保存の下準備」を推奨する。

理由:

- セーブ影響を先に閉じ込められる。
- UIや記録追加に入る前に、`idols` 状態の正規化とテストを固められる。
- 後続PRでライブ加算、UI表示、交流メモ解放を小さく積める。
