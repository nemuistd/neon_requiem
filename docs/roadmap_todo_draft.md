# NEON REQUIEM 今後の実装 TODO ドラフト

## この文書の位置づけ

この文書は、NEON REQUIEM の今後の実装候補を整理するための TODO ドラフトである。

確定仕様書ではない。実装と手動確認を進めながら、順番、内容、粒度を更新してよい。

既存の `docs/codex_tasks.md` は、これまで Codex に渡してきたタスク文の履歴として扱う。この文書は、今後の候補を見渡し、次に 1 タスクずつ切り出すための作業メモとして使う。

基本方針:

- 静的 Web アプリのまま進める。
- 既存の灯り獲得、施設強化、自動生産、保存、オフライン報酬を壊さない。
- 1 回の実装タスクは小さく、手動確認しやすい範囲にする。
- 新機能を追加する前に、それがアイドル、施設、歌、アイテム、記録、UI、保存のどれに属するかを明確にする。
- 複数リソース、ガチャ、課金、ログイン、バックエンド、クラウドセーブは当面扱わない。

## 完了済み

### アイドルタブの最小実装

- 右側コンテンツ領域にアイドルタブを追加した。
- 既存の `IDOLS` / `IDOL_ORDER` を使って、解放済み・未解放・解放条件・パッシブ効果を一覧表示する。
- 注目アイドル切替、保存、自動生産は維持する。

### 設定ボタンの最小機能化

- 右上の設定ボタンから小さな設定パネルを開ける。
- バージョン表示、閉じる、セーブデータ削除の最小機能を入れた。
- セーブデータ削除には確認を挟む。

### セーブデータリセット UI

- 設定パネル内からセーブデータを削除してリロードできる。
- `localStorage` の保存キー削除とリロードだけに限定した。

## 次に着手しやすい短期 TODO

### 記録追加

目的:

- 世界観を少しずつ開示し、施設や歌の解放に意味を持たせる。

実装メモ:

- 既存の `RECORDS` / `RECORD_ORDER` に追加する。
- 灯り、生産量、コストには影響させない。
- 解放条件は既存の施設 Lv または歌取得を使う。
- 序盤 UI では「祈念工学」「アンカー」などの裏側用語を出しすぎない。

完了条件:

- 新しい記録が条件達成で解放される。
- 既読状態が保存される。
- コンテンツ検証が通る。

### 歌アップグレードの追加とバランス調整

目的:

- 施設強化以外の恒久成長先を少し増やし、進行の節目を作る。

実装メモ:

- 歌は 1 回購入型の恒久アップグレードとして維持する。
- 追加する場合は 1 曲ずつにする。
- 効果は既存の `manualLightGainBonus` または `facilityProductionMultiplier` の範囲を優先する。
- 新しい効果型を追加する場合は、先に設計を確認する。

完了条件:

- 新しい歌が条件達成で解放される。
- 灯りを消費して取得できる。
- 取得後の効果が UI と数値に反映される。
- 既存歌の挙動が変わっていない。

### オフライン報酬上限の検討

目的:

- 長時間放置後に灯りが増えすぎる可能性を抑えるかどうか判断する。

実装メモ:

- まずは実装せず、上限が必要かを検討するタスクとして扱う。
- 上限候補は 8 時間、12 時間、24 時間など。
- 上限を入れる場合は、UI メッセージに違和感が出ない文言も同時に考える。

完了条件:

- 上限を入れるか、当面入れないかを決める。
- 入れる場合は、上限秒数と手動確認方法が明確になる。

### モバイル表示の軽い調整

目的:

- 小さい画面でも、上部操作、メッセージ、注目アイドル、主要タブを見失いにくくする。

実装メモ:

- 本格的なモバイル専用 UI は作らない。
- 既存 CSS のレスポンシブ調整に留める。
- 文字の重なり、ボタンの押しにくさ、横スクロールを優先して直す。

完了条件:

- スマホ幅でも主要操作が画面内に収まる。
- テキストがボタンやカードからはみ出さない。
- 既存のデスクトップ表示が大きく崩れていない。

## 仕様検討が必要な中期 TODO

### アイテムシステム

目的:

- 歌より小〜中規模の恒久アップグレードとして、通常成長の選択肢を増やす。

検討メモ:

- アイテムは施設や歌と役割が重ならないようにする。
- 効果は小さめにし、複雑な装備やインベントリにはしない。
- UI では歌と同じタブに置くか、別タブにするかを先に決める。

完了条件:

- アイテムの役割、保存形式、UI 表示、最初の 1〜3 個の候補が決まる。

### アイドルごとの交流イベント

目的:

- アイドルを単なる倍率要素にせず、居場所づくりや復興の手触りを増やす。

検討メモ:

- 交流は搾取や管理ではなく、支援、会話、復興への参加として扱う。
- 複雑なストーリー分岐や好感度システムはまだ入れない。
- 最初は条件達成で短い会話が読める程度がよい。

完了条件:

- 交流の発生条件、表示場所、保存する状態が決まる。

### アイドル詳細画面

目的:

- 注目アイドル枠に収まりきらないプロフィール、解放条件、関係する記録を整理する。

検討メモ:

- 詳細ボタンを既存 UI から使えるようにする。
- 最初は詳細モーダルまたは右側アイドルタブ内の詳細表示でよい。
- 新しい効果や交流イベントを同時に追加しない。

完了条件:

- 詳細画面の表示範囲と閉じ方が決まる。
- 既存データだけで最低限の表示が成立する。

### 新区画・新施設

目的:

- 復興対象を増やし、地下都市が広がっていく感覚を作る。

検討メモ:

- 施設は原則として灯りを直接生産する。
- 全体倍率だけを主機能にする施設は避ける。
- 新施設は、記録やアイドル解放などの意味とセットで追加する。

完了条件:

- 施設名、解放条件、基本コスト、コスト倍率、生産量、関連記録が決まる。

### 新アイドル追加

目的:

- 新区画や記録と結びついた新しい灯し手を追加する。

検討メモ:

- アイドル追加は単なる倍率追加にしない。
- 肩書き、担当する雰囲気、解放条件、関連記録を同時に考える。
- 画像追加は別タスクとして扱ってもよい。

完了条件:

- 名前、読み、肩書き、説明、解放条件、パッシブ効果が決まる。

### 復旧報告・観測記録の拡充

目的:

- 霞、灯し手、祈念工学などの裏側設定を、少しずつ安全に開示する。

検討メモ:

- 序盤では不穏な用語を出しすぎない。
- 旧施設ログでは「アンカー」「観測者」「局所安定化」などを使ってよい。
- 通常 UI では「聖歌」をシステム名にしない。

完了条件:

- 追加する記録のタイトル、カテゴリ、本文、解放条件が決まる。

### ゲームバランス調整方針

目的:

- 進行速度、施設コスト、歌コスト、オフライン報酬の感触を整える。

検討メモ:

- まずは現状の到達時間を手動確認する。
- 数値変更はまとめて大きく変えず、目的ごとに小さく行う。
- 変更前後の確認手順を残す。

完了条件:

- 調整対象、理由、期待するプレイ時間、確認方法が決まる。

## まだ実装しないもの

以下は、明示的に必要になるまで TODO に昇格しない。

- 複数リソース
- ガチャ
- 課金
- ログイン
- バックエンド
- クラウドセーブ
- 転生 / プレステージ
- 音声
- 複雑なストーリーシステム
- 複雑なアイドル編成
- インベントリ管理
- 大規模な画像制作
- 本格的なモバイル専用 UI

理由:

- 現在の強みである、小さく遊べて保存できる静的ブラウザゲームという形を崩しやすい。
- 保存形式、UI、バランスへの影響が大きい。
- 先に既存ループと表示を磨いた方が、後続の仕様判断がしやすい。

## 優先順位の考え方

優先度が高いもの:

- 既存の未実装ボタンや未実装タブを自然に埋めるもの。
- 既存データを活用でき、保存形式の変更が小さいもの。
- プレイヤーが次に何をすればよいか分かりやすくなるもの。
- 手動確認が短時間でできるもの。

優先度を下げるもの:

- 新しい保存形式や移行が大きいもの。
- UI 全体の再設計が必要なもの。
- バランス調整なしでは成立しないもの。
- MVP の静的 Web アプリ方針を崩すもの。

現時点のおすすめ順:

1. 記録追加
2. モバイル表示の軽い調整
3. 歌アップグレードの追加とバランス調整
4. アイドル詳細画面の検討
5. アイテムシステムの検討

## Codex に渡すときの注意

Codex に依頼するときは、必ず 1 タスクずつ渡す。

依頼文に含めるとよいもの:

- 変更対象の TODO 名。
- 実装する範囲。
- 今回は実装しないもの。
- 変更後に報告してほしい内容。
- 手動確認してほしいシナリオ。

依頼例:

```text
docs/roadmap_todo_draft.md の「アイドルタブの最小実装」だけを実装してください。
新しいアイドル効果、交流、編成システムは追加しないでください。
実装後、変更ファイル、実行したコマンド、手動確認方法、意図的に実装しなかったものを報告してください。
```
